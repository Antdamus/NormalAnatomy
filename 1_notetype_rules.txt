-----------------------------------------------------

NORMAL ENGINE — SHARED CHASSIS RULE

This engine MUST reuse the same note type, field order, TSV structure, media pipeline,
ID logic, and tag architecture as the pathology engine.

Do NOT invent new fields.
Do NOT reorder fields.
Do NOT create a second note type unless the user explicitly decides to do so.

The normal engine changes FIELD USAGE, not infrastructure.

-----------------------------------------------------

FIELD-USAGE PRINCIPLE

A field may be:
• REQUIRED
• OPTIONAL
• CONDITIONALLY ALLOWED
• NORMALLY BLANK
• FORBIDDEN

Do NOT force content into a field just because the field exists.

If a field does not naturally fit a normal-anatomy / normal-variant learning goal,
leave it blank.

Blank is preferred over fake content.

-----------------------------------------------------

GLOBAL NORMAL-ENGINE FIELD INTERPRETATION

In the normal engine, the note fields are repurposed as follows:

• Question
  = normal-structure, normal-relationship, normal-appearance, or normal-vs-variant prompt

• Entity_Label
  = target structure, target anatomic concept, target normal variant, or target pitfall entity

• Imaging_Differentiation
  = why this is normal, why this is a variant, or what separates it from pathology

• Boards_Trap
  = classic normal mimic, pseudolesion, look-alike, or interpretive trap

• High_Yield_Q / High_Yield_A
  = definition, landmark rule, modality-specific normal concept, or compact anatomic teaching point

The normal engine should NOT force pure normal material into:
• Most_Likely_Diagnosis
• Differentials
• Mechanism

unless the specific card subtype genuinely requires it.

-----------------------------------------------------

FIELD RULES — NORMAL ENGINE

1) Clinical_Context
REQUIRED ALWAYS

Because Clinical_Context can never be empty, every normal-engine card must populate it.

Allowed forms:

A) Image-based normal case / anatomy-recognition card
• Short
• Neutral
• Non-diagnostic
• Can include age/sex if source-supported
• Can include modality/exam context ONLY when needed for normal anatomy learning and not answer leakage
• Must remain unique at the note level

B) Standalone concept / landmark / variant card
• Unique 12-character alphanumeric/symbolic code only
OR
• Another shared-system-compliant uniqueness format if the broader framework later standardizes one for standalone normal cards

Do NOT fabricate clinical history.

-----------------------------------------------------

2) Image
REQUIRED for image-based normal cards
OPTIONAL for concept-only normal cards

Use for:
• normal anatomy recognition
• normal relationship drills
• normal modality appearance
• normal variants
• pseudolesion / mimic cards

Do NOT populate Image just to imitate UNKNOWN architecture.

-----------------------------------------------------

3) Image_Annotated
OPTIONAL but strongly encouraged when labels materially improve learning

Use for:
• labeled anatomy
• relationship drills
• landmark teaching
• variant localization

If populated:
• must remain verbatim-caption compliant
• must preserve image order exactly
• must not add unsupported interpretation

-----------------------------------------------------

4) Question
REQUIRED for image-based recognition cards
OPTIONAL for some standalone concept cards when another card type controls the front

Allowed question goals:
• identify the structure
• identify the normal relationship
• identify the normal variant
• identify the expected normal appearance
• decide normal vs variant vs abnormal
• name the feature preventing overcall as disease

Question must NOT:
• force disease-first diagnosis logic for a pure normal card
• include answer leakage
• include unsupported interpretation

Preferred examples:
• “Identify the labeled structure.”
• “What normal anatomic relationship is demonstrated?”
• “Is this normal anatomy, a normal variant, or abnormal?”
• “What normal feature prevents misdiagnosis here?”

-----------------------------------------------------

5) Most_Likely_Diagnosis
NORMALLY BLANK

FOR PURE NORMAL CARDS:
• leave blank

ALLOWED ONLY WHEN:
• the card intentionally tests rejection of a pathology mimic
AND
• the card design truly requires a diagnosis-style answer field

Do NOT insert fake answers such as:
• “normal study”
• “no diagnosis”
unless that exact wording is the true source-supported teaching target

-----------------------------------------------------

6) Entity_Label
VERY IMPORTANT IN NORMAL ENGINE

Use Entity_Label to store the true learning target:
• structure name
• anatomic region
• named normal variant
• named normal pitfall
• physiologic state
• developmental appearance
• normal imaging concept

Entity_Label replaces much of the disease-label role it plays in pathology workflows.

Do NOT overload Entity_Label with long reasoning text.

-----------------------------------------------------

7) Differential_Q
NORMALLY BLANK

Allowed only for:
• normal-vs-abnormal discrimination drills
• normal-variant-vs-pathology discrimination drills
• tightly bounded mimic questions

Do NOT use Differential_Q for routine normal anatomy cards.

-----------------------------------------------------

8) Differentials
NORMALLY BLANK

Allowed only when the actual educational goal is discrimination between:
• normal and abnormal
• normal variant and pathology
• pseudolesion and true lesion

If used:
• keep tightly restricted
• include only realistic competing entities
• do not inflate into pathology-deck behavior by default

-----------------------------------------------------

9) Imaging_Differentiation
HIGH-VALUE FIELD IN NORMAL ENGINE

Use for:
• why this is normal
• what makes the relationship normal
• what keeps the appearance from being overcalled
• what feature distinguishes the variant from pathology
• modality-specific separator points

This field is often more important in the normal engine than in the pathology engine.

-----------------------------------------------------

10) Original_Caption
Populate only when required by image/template logic.

Do not paraphrase.
Do not modify verbatim source captioning rules.

-----------------------------------------------------

11) Mechanism_Q
NORMALLY BLANK

Allowed only when the card genuinely teaches:
• developmental basis
• physiologic basis
• modality-appearance basis
• structural reason for expected appearance

Do NOT use Mechanism_Q for routine anatomy naming.

-----------------------------------------------------

12) Mechanism
NORMALLY BLANK

Allowed only when a true explanatory mechanism materially improves understanding, such as:
• physiologic explanation for normal appearance
• developmental explanation for variant appearance
• anatomic explanation for expected modality pattern

Do NOT force pathophysiology-style essays into normal cards.

-----------------------------------------------------

13) Boards_Trap_Q
OPTIONAL

Very useful for:
• normal mimics of disease
• pseudolesions
• normal variants commonly overcalled
• misleading appearances on specific modalities

Should ask for the trap and/or the safety check.

-----------------------------------------------------

14) Boards_Trap
OPTIONAL

Very useful in this engine.

Use for:
• classic overcall traps
• pseudo-masses
• normal folds, interfaces, vessels, ossicles, contours, artifacts, or physiologic states that mimic pathology
• the key maneuver, feature, or discriminator that keeps the reader safe

-----------------------------------------------------

15) High_Yield_Q
OPTIONAL

Use for:
• definition cards
• normal landmark rules
• modality-specific normal concepts
• compact structure/relationship anchors
• variant definitions

-----------------------------------------------------

16) High_Yield_A
OPTIONAL

Use for:
• compact source-faithful definitions
• landmark logic
• normal measurement rules
• short normal-vs-variant boundary statements
• anatomy-first teaching points

Do not turn High_Yield_A into an unstructured paragraph dump.

-----------------------------------------------------

17–20) Radiopaedia fields
Leave blank unless intentionally generating the Radiopaedia-specific workflow later.

Do not populate these fields casually.

-----------------------------------------------------

21) summary
Follow the same global summary rule as the shared system.

If the card originates from RadPrimer / STATdx:
• summary is REQUIRED
• must reflect full-source review
• must not mix in Core or outside knowledge

If the card does NOT originate from RadPrimer / STATdx:
• leave summary blank

-----------------------------------------------------

DEFAULT PURE NORMAL CARD SHAPE

For a pure normal anatomy / normal appearance card, the usual pattern is:

• Clinical_Context = populated
• Image = populated if image-based
• Image_Annotated = optional
• Question = populated
• Most_Likely_Diagnosis = blank
• Entity_Label = populated
• Differential_Q = blank
• Differentials = blank
• Imaging_Differentiation = optional
• Original_Caption = conditional
• Mechanism_Q = blank
• Mechanism = blank
• Boards_Trap_Q = optional
• Boards_Trap = optional
• High_Yield_Q = optional
• High_Yield_A = optional
• Radiopaedia fields = blank unless intentionally used
• summary = source-dependent per global rules

-----------------------------------------------------

DEFAULT NORMAL VARIANT / PITFALL CARD SHAPE

For a normal variant / mimic / pitfall card, the usual pattern is:

• Clinical_Context = populated
• Image = populated if image-based
• Question = populated OR Boards_Trap_Q populated, depending on card type
• Entity_Label = populated
• Imaging_Differentiation = strongly encouraged
• Boards_Trap_Q / Boards_Trap = strongly encouraged
• Differentials = allowed only if true discrimination is being taught
• Most_Likely_Diagnosis = usually blank
• Mechanism = only if genuinely explanatory
• summary = source-dependent per global rules

-----------------------------------------------------

FIELD HYGIENE RULE

Never populate a field with:
• filler
• repetition
• fake differentials
• fake diagnoses
• pseudo-mechanisms
• unsupported variant claims
• “N/A”

Blank is better than false structure.

-----------------------------------------------------

COMPATIBILITY RULE

All normal-engine outputs must remain fully compatible with:
• shared TSV export
• shared import workflow
• shared media references
• shared tags
• shared automation pipeline

The engine changes card logic, not the chassis.
