# Guided Image Mastery Narrative Prompt

You are transforming a radiology source article into a **GUIDED IMAGE MASTERY NARRATIVE** for a radiology resident.

This output is designed for **fast, high-yield first-pass learning** when time is limited.

It is **not** primarily a flashcard replacement.
It is **not** a trivia quiz stream.
It is **not** a generic summary.
It is **not** entertainment-only storytelling.

The narrative must do four things at once:

1. **Reorganize dry article content into a coherent teaching sequence**
2. **Force deliberate inspection of the extracted images**
3. **Preserve the important educational payload of the article**
4. **Build a strong conceptual framework that Anki can later reinforce**

---

## PRIMARY EDUCATIONAL ROLE

This guided narrative is the **understanding + visual organization layer**.

The main recall burden should still remain in **Anki**.

Therefore:

* The narrative should include **targeted pauses and brief prompts**
* But it should **not** become a full barrage of detailed recall questions
* The narrative should be optimized for:

  * comprehension
  * pattern organization
  * visual attention
  * discriminative noticing
  * case-to-case comparison
  * retention of major pivots

Think of this as:

**Narrative = guided understanding + image walkthrough + structure**
**Anki = deeper retrieval practice + durable memory consolidation**

## INPUT FORMAT EXPECTATION

You will receive an extracted block that may contain sections such as:

* `=== TOPIC ===`
* `=== CORE VALIDATION INPUT ===`
* `=== PROMPT ===`
* `=== ARTICLE ===`
* `=== IMAGES (selected; original numbering preserved) ===`

The article section contains the structured content outline.
The image section contains the extracted image labels, filenames, and captions.

Possible image labels:

* `IMAGE_01`, `IMAGE_02`, etc.
* `CASE_01`, `CASE_02`, etc.
* source-qualified fused-source labels such as `RadPrimer image 5`, `STATdx image 35`, `RP-05`, or `SDX-35`

These labels do **not** always mean the same thing.

### IMPORTANT IMAGE / CASE INTERPRETATION RULE

If the extractor output uses `CASE_##` because explicit grouping was provided, you may treat that as a true grouped case.

If the extractor output uses `IMAGE_##`, you must **not** assume those are already true cases.

However, you may attempt **controlled grouping inference** from the captions.

For the spoken narrative itself, convert image labels into **speech-friendly references**.

Examples:

* `IMAGE_01` -> `image one`
* `IMAGE_02` -> `image two`
* `IMAGE_14 through IMAGE_19` -> `images fourteen through nineteen`
* `RadPrimer image 5 / RP-05` -> `RadPrimer image five`
* `STATdx image 35 / SDX-35` -> `STATdx image thirty five`

Do not read image labels aloud with underscores or leading zeros in the Speechify narrative.

For fused RadPrimer plus STATdx source packages, do **not** say both the long source label and the short code in the spoken narrative.

Correct:

* `STATdx image thirty five`
* `RadPrimer image five`

Incorrect:

* `STATdx image thirty five, SDX thirty five`
* `RadPrimer image five, RP five`

Keep short labels such as `SDX-35` and `RP-05` only for internal accounting, download filenames, card metadata, audit blocks, or traceability. They should not be read aloud in the lecture when the long source-qualified image reference is already present.

Also normalize speech-confusing modality abbreviations in the spoken narrative.

Examples:

* say `MRI` instead of `MR`
* say `CT` as written
* prefer speech-friendly phrasing such as `T1-weighted MRI` rather than abbreviated strings that text-to-speech may mangle

Do not use bare `MR` in the Speechify narrative, because text-to-speech may pronounce it as `mister`.

Preserve the exact numeric accounting internally for the grouping audit code block, but make the spoken walkthrough sound natural to a listener.

## CONTROLLED GROUPING INFERENCE RULE

When preparing the guided narrative, you may infer likely multi-image case groupings from caption language such as:

* “in the same patient”
* “same patient”
* “same case”
* modality progression across sequential images
* direct cross-reference to a prior image
* explicit continuation language

### Grouping rules

1. If grouping confidence is **high**, you may narrate the images together as a likely grouped case.
2. If grouping confidence is **moderate**, you must explicitly label the grouping as **inferred**.
3. If grouping confidence is **low**, keep the images separate.
4. Do **not** state inferred grouping as certain unless the extractor explicitly grouped it.
5. When uncertain, prefer language like:

   * “This image likely pairs with the prior image.”
   * “These next two images appear to belong to the same patient.”
   * “This is likely an inferred multi-image case.”
6. Do **not** silently merge many images into a single case without saying so.
7. If a pathology photograph or gross specimen is being linked to an imaging case by caption continuity rather than an explicit same-patient statement, keep the wording cautious and do not overstate certainty.

### Transparency rule

If you group images by inference, you must explicitly acknowledge that the grouping is inferred from caption language.
This should normally happen once near the beginning of the narrative, not as a repeated local disclaimer.

## GLOBAL GROUPING TRANSPARENCY RULE

If any multi-image groupings are inferred from caption continuity rather than explicitly provided by the extractor:

* acknowledge that once near the beginning of the narrative in a compact orientation note
* after that, proceed naturally with the grouped walkthrough
* re-mention uncertainty locally only if a particular grouping is especially weak or ambiguous
* do not repeatedly insert the same inferred-grouping disclaimer throughout the narrative
* avoid repetitive phrasing such as:

  * “these likely belong together”
  * “this is an inferred grouping”
  * “this grouping is inferred”

unless that local uncertainty genuinely matters

## IMAGE COMPLETENESS RULE

If an image section is provided, every provided image must be accounted for.

This is mandatory.

* Do not skip images.
* Do not silently exclude images because they seem repetitive, low-yield, poorly captioned, or hard to integrate.
* Every provided image number must appear in the narrative walkthrough, either individually or as part of an explicitly stated grouped case.
* Every provided image number must also appear in the final `INCLUDE` array.
* If an image has limited educational value, handle it briefly, but still tell the learner to look at it and state why it is being handled briefly.
* If an image has no caption, walk through it conservatively and explicitly say that the image is not richly captioned; do not invent findings.
* If an image cannot be interpreted safely from the supplied material, still account for it and explain that the source does not support a detailed teaching point.

Only omit an image if the user explicitly says to exclude it.

## ARROW / ANNOTATION VERBALIZATION RULE

Many image captions contain inline arrow image tags such as `<img src="arrow_WO.png">`.

These arrow tags are not decorative. They are the source-supported bridge between the image and the finding.

When walking through any image or grouped case, you must explicitly verbalize the caption-supported arrows/callouts so the learner can follow the image without reading the caption separately.

Use speech-friendly language such as:

* "the white open arrow marks..."
* "the white solid arrow points to..."
* "the black open arrow identifies..."
* "the cyan solid arrow highlights..."

Common arrow code meanings:

* `WO` = white open arrow
* `WS` = white solid arrow
* `WC` = white curved arrow
* `BO` = black open arrow
* `BS` = black solid arrow
* `BC` = black curved arrow
* `CO` = cyan open arrow
* `CS` = cyan solid arrow
* `CC` = cyan curved arrow

Rules:

* Do not read the HTML tag or filename aloud.
* Do not say only "the arrow" when multiple arrows are present; distinguish the arrows by color/style when the code allows it.
* Integrate arrow callouts into the teaching flow instead of dumping the caption.
* If multiple arrows are present, describe them in the order that best supports image inspection: usually primary finding first, then associated findings, complications, anatomy, or comparison points.
* If the caption has arrow tags but the surrounding caption text does not clearly state what an arrow marks, do not invent a finding. Use cautious language such as "the marked callout corresponds to the caption-described finding."
* If an image has arrows, the local image explanation should usually name what each arrow/callout is showing before moving on to broader article facts.

The goal is that during audio playback the learner hears exactly how the arrows connect to the visual findings.

## IMAGE ORDER FLEXIBILITY RULE

The narrative must be image-complete, but it does NOT have to preserve the raw image order.

After choosing the organizing spine, arrange the image walkthrough in the order that best supports teaching and retention.

Allowed image-ordering strategies include:

* subtype or diagnostic bucket order
* mechanism-to-imaging order
* modality order
* severity / complication order
* classic pattern first, then variants and mimics
* benign-to-aggressive or low-risk-to-high-risk progression
* diagnostic workflow order
* timeline / disease evolution order

Rules:

* Do not force image one, then image two, then image three if that weakens the teaching structure.
* If the image order is changed, make the transitions clear so the learner understands why the next image appears now.
* Keep true same-patient or same-case groups together unless there is a strong teaching reason to split them.
* Do not split an explicitly grouped case without explaining why.
* If images are reordered, still name every provided image in the narrative and include every provided image number in the final `INCLUDE` array.
* The final grouping audit still uses the original numeric image numbers, even if the spoken narrative walks through them in a different order.
* Reordering must improve structure; do not reorder randomly or just for novelty.

The best output should feel like:

**The image order serves the teaching framework; the final audit preserves the original image accounting.**

## CORE DESIGN PRINCIPLE

The narrative must function as a **guided walkthrough**.

It should repeatedly do things like:

* orient the learner
* point them to a specific image or image cluster
* tell them what to inspect
* explain why it matters
* connect the image back to the larger concept

This is not just “summarize the article.”
It is “teach the article while walking the learner through the images.”

## ORGANIZING SPINE RULE

Before writing the narrative, first infer the article's **best organizing spine**.

The organizing spine is the main structure that makes the topic easiest to understand and remember.

Choose the structure that best fits the source, such as:

* fracture pattern or morphology
* disease subtype or diagnostic bucket
* modality
* severity or aggressiveness
* mechanism or timeline
* differential branches
* diagnostic workflow
* anatomy-based distribution

Then rebuild the narrative around that spine rather than around the raw article order, unless the source article order is already clearly optimal.

Rules:

* name the major buckets early in the opening orientation or big-picture map
* make the learner feel that the topic has a clean internal architecture
* do not let the narrative become a sequence of detached pearls before the organizing framework is clear
* if the article contains many facts, classifications, and cases, the narrative must still present them through the chosen structure rather than as serial accumulation
* prefer approximately 3-6 major buckets when the source supports that range
* if the first pass creates too many weak buckets, merge them into stronger, more memorable categories
* do not create a taxonomy that is more complex than the article actually needs

If the source supports more than one possible structure, prefer the one that best improves:

* recognition
* discrimination
* comparison across cases
* later retrieval in Anki

---

## CASE-TO-STRUCTURE INTEGRATION RULE

Cases and images must serve the organizing spine, not interrupt it.

When introducing a case or grouped image set, make clear why it belongs where it does. For example:

* it is a classic example of bucket A
* it contrasts with bucket B
* it shows progression from one pattern to another
* it demonstrates a pitfall inside the current framework
* it refines or stress-tests the current category

Do not let cases feel like isolated stops on a tour.

The learner should feel that each case strengthens the structure already being built.

---

## LOCAL CONTINUITY / ANTI-CHUNKING RULE

The narrative should not feel like a series of separate image reports.

After the organizing spine is chosen, every image or image cluster should have a clear job inside that spine:

* establish the classic pattern
* explain the mechanism behind the appearance
* show a modality-specific version of the same concept
* introduce a variant, mimic, complication, or management pivot
* contrast with the immediately prior image or prepare the next image

When moving from one image or cluster to the next, use bridge logic whenever the source supports it.

Preferred transitions should sound like:

* "Now that you have seen the classic pattern, move to image..."
* "This next case stress-tests the same rule because..."
* "Compare that with the next image, where the same disease behaves differently..."
* "The prior image showed the diagnosis; this one shows why it matters clinically..."

Avoid repetitive block structure where every image begins, develops, and ends as an isolated mini-lecture.

If several images belong to the same diagnostic bucket, first name the bucket, then let the images build one local arc inside that bucket. Do not reset the whole topic for each image.

If an image truly cannot be integrated because the caption is sparse or disconnected, handle it briefly and say why it is a limited standalone teaching point.

---

## NO ORPHAN FACTS RULE

Do not output detached fact clusters.

Any important non-image fact should be attached to one of the following:

* a diagnostic bucket
* a case comparison
* a mechanism explanation
* a modality comparison
* a reporting/management pivot
* a closeout synthesis point

Before including a fact, implicitly ask:

* Why is this fact here?
* What part of the framework does it support?
* Does it change recognition, differential, severity, prognosis, or reporting?

If it does not strengthen the framework, omit it unless the source makes it essential.

---

## CONTENT-IN-CONTEXT RULE

Do not create isolated lecture chunks of article content.

The best place for a fact is usually the moment when an image, image cluster, bucket transition, or comparison makes that fact useful.

Examples of good placement:

* introduce mechanism immediately after an image shows the appearance that mechanism explains
* introduce epidemiology when the patient or risk setting in a case makes that epidemiology meaningful
* introduce pathology when the image appearance needs a tissue-level explanation
* introduce differential diagnosis when an image could plausibly be mistaken for a mimic
* introduce management or prognosis when an image shows size, complication, aggressive behavior, distribution, or another decision-changing feature
* introduce modality strengths when moving from one modality or phase to another

Avoid long standalone blocks such as:

* a separate epidemiology dump after the image walkthrough
* a separate pathology dump disconnected from the image appearance
* a separate management dump when no case or imaging feature has created the need for it
* a differential list that is not tied to a visual branch point or imaging discriminator

If important content has no natural image anchor, place it compactly in the opening map, a transition between buckets, cross-image synthesis, or high-yield closeout. Even there, frame it as part of the organizing structure rather than as a loose list.

Use between-image transitions as teaching space. A good transition can carry one or two article facts when those facts explain why the next image matters.

Before finalizing, scan for any paragraph that could be removed without changing the image walkthrough or organizing framework. If such a paragraph exists, either integrate it into the relevant image/bucket or compress it into the closeout.

## NARRATIVE STRUCTURE (MANDATORY)

Unless the user requests another format, organize the output in this order:

### 1. Opening orientation

Give a concise but strong opening that tells the learner:

* what the topic is
* why it matters
* what the article is mainly trying to teach
* what the major learning pivots will be

Keep this compact but meaningful.
In most cases, keep the opening to no more than two short paragraphs before moving into the structural map.
Do not spend so long in the opening that visual engagement is delayed.

### 2. Big-picture map

Before diving into image walkthroughs, briefly organize the topic into major buckets such as:

* classic imaging pattern
* modality-specific appearance
* variant appearances
* aggressive features
* mimics/differentials
* clinical or reporting pivots

This should make the dry article feel navigable.

This map is not filler.
It should declare the organizing spine that will control the rest of the narrative.

Whenever possible:

* make the buckets mutually clarifying rather than redundant
* present them in the order that best supports understanding
* keep the map concise but structurally strong enough that later cases clearly fit into it

Use a macro-before-micro flow whenever the source supports it:

* broad pattern or category first
* then subtype, contrast, or branch point
* then case-level detail
* then pearl, pitfall, or reporting implication

For each major bucket, prefer the following micro-template when the source supports it:

* what this bucket is
* what the images in this bucket demonstrate
* what separates this bucket from the nearest mimic, neighboring subtype, or competing branch
* one-line takeaway

### 3. Guided image walkthrough

This is the core of the output.

If images are provided, this section is mandatory and must account for every provided image.

The walkthrough order should follow the chosen organizing spine, not automatically the raw extracted image order.

If raw image order is already the clearest teaching order, use it.
If another order teaches better, reorder the image or image clusters deliberately.

For each image or inferred image cluster:

* explicitly identify the image label(s)
* tell the learner to look at it
* tell them what to notice first
* tell them what the dominant teaching point is
* connect that image to the underlying concept
* compare it with prior images when useful
* mention if the grouping is inferred rather than explicit

Do not skip any provided image. If several images are repetitive, group them when justified, but still name every image in the group and briefly state the shared teaching point.

When the walkthrough moves from one major bucket to another:

* explicitly signal the transition
* make clear why the next bucket is distinct from the prior one
* preserve momentum so the learner feels guided through one coherent framework rather than reset into a new mini-lecture

### 4. Cross-image synthesis

After the walkthrough, summarize what patterns repeated across the images, such as:

* recurring hallmark features
* modality-dependent changes in appearance
* features of aggressiveness
* clues against major mimics
* what keeps showing up across examples

This section should synthesize what the learner already saw.
Do not use cross-image synthesis as a place to introduce a long new block of unrelated clinical, pathology, management, or epidemiology content.
If a remaining source fact must be included here because it had no natural image anchor, connect it explicitly to the repeated image pattern or to the organizing spine.

### 5. High-yield closeout

End with a compact high-yield teaching finish that includes:

* major diagnostic pivots
* major mimics or traps
* the most testable / report-relevant ideas
* what the learner should now be able to recognize confidently

The closeout must remain inside the same organizing spine established earlier.
Do not let the ending relapse into detached lecture notes or a loose checklist that ignores the framework used in the main walkthrough.
Whenever possible, close by briefly revisiting the same major buckets in compressed form rather than inventing a new end-stage structure.
Do not save major article content for the closeout if it could have been taught more naturally at the relevant image, transition, or bucket.

## IMAGE WALKTHROUGH RULES

For every provided image or inferred grouped case, the narrative should usually follow this sequence:

Every provided image must be walked through. The walkthrough may be compact for repetitive or low-information images, but it must still explicitly identify the image and give the learner a reason to inspect it.

1. **Orientation**

   * “Now go to image seven.”
   * “Now look at images fourteen through nineteen; these likely belong together based on caption continuity.”

2. **Inspection instruction**

   * Tell the learner what to inspect first
   * Example categories:

     * lesion enhancement behavior
     * vascular invasion
     * fat content
     * capsule
     * washout
     * diffusion restriction
     * liver background
     * multiplicity
     * extrahepatic spread

3. **Meaning**

   * Explain why the finding matters
   * Tie it to diagnosis, aggressiveness, mimic separation, or modality understanding

4. **Comparison**

   * Compare with earlier or later images when useful
   * Example:

     * CT vs MR
     * noncirrhotic vs cirrhotic liver
     * solitary vs multifocal
     * viable vs nonviable tumor

5. **Micro-pause**

   * Occasionally prompt the learner briefly
   * Example:

     * “Pause here and make yourself say the dominant clue out loud.”
     * “Pause and decide what makes this more than a generic hypervascular liver lesion.”

These micro-pauses should be selective and efficient.

## QUESTION DENSITY RULE

Do **not** overload the narrative with constant questioning.

Use **brief guided prompts**, not nonstop interrogation.

Preferred question style:

* “What is the dominant clue here?”
* “What would you not want to miss in this image?”
* “What makes this different from a mimic?”
* “What feature raises this from suspicious to strongly characteristic?”

Avoid long multi-step quiz blocks unless the user explicitly requests them.

The narrative should feel like:

* guided attention
* active noticing
* occasional self-check

—not a flashcard deck read aloud.

## WHEN TO EXPLICITLY TELL THE LEARNER TO LOOK AT THE IMAGE

You should explicitly direct the learner to the image whenever:

* a new image or grouped image set begins
* the teaching pivot depends on a visual clue
* modality comparison matters
* a major discriminator is being taught
* a trap or mimic is being separated
* an aggressive feature is being demonstrated

Good examples:

* “Now look closely at image three.”
* “Stay on this image for a moment.”
* “Now compare this with the prior image.”
* “Now move to image seventeen.”
* “These next images likely form one inferred case sequence; look at them in order.”

## DRY ARTICLE RECONSTRUCTION RULE

The article may be dry, repetitive, or organized like a reference outline.

Your job is to transform it into a **coherent teaching arc**.

That means you should:

* merge related points into cleaner teaching units
* pull forward the most important pattern-recognition pivots
* connect pathology/clinical information back to the imaging
* reduce unnecessary repetition
* preserve important details and discriminators
* reorganize the material into the best teaching architecture supported by the source
* make the governing framework clear before accumulating details
* convert loose lists into buckets, contrasts, progressions, or decision branches whenever the source supports that move
* preserve important details, but place them after the learner already knows the bucket they belong to
* move from macro pattern to micro detail whenever the source supports that sequencing

Do **not** simply read the article outline back verbatim.
Do **not** preserve a weak source order if a clearer, more memorable order is available from the same source.

This includes facts such as:

* important numbers
* modality performance data
* memorable epidemiology
* pathology/genetics
* staging/prognosis
* treatment/follow-up facts

Rules:

* attach them to the relevant image discussion, modality comparison, disease progression explanation, recurrence discussion, or closeout synthesis
* prefer contextualized integration over isolated listing
* make the integration feel seamless, elegant, and cumulative rather than bolted on
* the narrative should not feel like image discussion plus random fact insertion
* it should feel like one continuous teaching script in which images, modality reasoning, numbers, performance data, pathology, prognosis, and clinical pivots reinforce each other
* avoid dumping disconnected facts
* no source-emphasized non-image fact may be omitted just because it lacks a clean image anchor
* if a fact is important but has no natural image anchor, integrate it into the opening map, cross-image synthesis, or high-yield closeout instead of creating a separate fact list
* whenever possible, place the fact at the exact moment where it helps distinguish one bucket from another or clarifies why the current case belongs in its category
* if several facts belong to the same bucket, compress them into a clean local synthesis rather than scattering them as trivia fragments
* preserve smaller but still meaningful article details when they improve specificity, such as classic demographics, favored location/distribution, subtype frequency, named contrast behavior, laboratory negatives, multiplicity thresholds, or uncommon-but-explicit imaging clues
* do not drop an article-supported detail merely because it is less dramatic than hemorrhage, rupture, or malignancy if it still helps the learner reconstruct the topic accurately
* when a detail is lower-stakes but still educationally useful, integrate it briefly rather than omitting it entirely

When finishing a major bucket, provide a brief one-line takeaway that states the core recognition or discrimination lesson of that bucket before moving on.

## IMAIOS ANATOMY LABEL OUTPUT

After the Speechify narrative and image grouping audit, also produce organized IMaios anatomy label blocks.
These blocks are for the user's IMaios engine, so each code block must be copy-paste ready.

Core rules:

* do not create one giant undifferentiated anatomy list
* organize labels into the order a learner should study the anatomy visually
* each IMaios block must represent one teachable anatomical chunk
* put the chunk title outside the code block
* inside each code block, output only labels, one label per line
* inside each IMaios code block, do not use bullets, numbering, explanations, comments, headings, or blank-line prose
* choose labels that are directly needed to understand the pathology, complications, routes of spread, surgical danger zones, and image findings
* prefer structures that can be found visually in IMaios over abstract disease concepts
* avoid disease names, findings, maneuvers, diagnoses, and teaching comments inside IMaios code blocks
* preserve practical visual relationships: region first, pathway next, then small parts, danger landmarks, spread routes, and differential traps

Anatomy-aware chunk sizing:

* do not group labels merely because they are anatomically related
* each block should be the smallest useful visual learning pass for that region and pathology
* prefer 3-7 labels per block, but use anatomy and visual review logic rather than a rigid count
* if a candidate block grows beyond about 8 labels, split it into anatomy-aware subchunks unless the labels form one uninterrupted named chain, pathway, route of spread, or procedural corridor
* choose the split order that makes sense for the local anatomy, such as framework first, then spaces or compartments, pathways, small parts, danger landmarks, routes of spread, and variants or mimics
* for temporal bone, natural subchunks may include outer bony framework, mastoid and middle ear air spaces, ossicular chain, otic capsule and labyrinth, facial nerve canal, skull base vascular landmarks, and normal fissures or pseudofracture lines
* for spine, natural subchunks may include vertebral body and endplates, posterior elements, facets, ligaments, canal and foramina, and paraspinal soft tissues
* for abdomen or pelvis, natural subchunks may follow organ, vascular pedicle, duct or lumen, compartment, peritoneal reflection, and danger landmark
* if a large region is split, make every subchunk its own human code block and its own JSON chunk
* use chunk titles that show the larger region and the subchunk focus, but keep only actual labels inside the code block
