Do not collapse a normal-anatomy article into pathology-style output just because it mentions disease mimicry.

If the source is still fundamentally teaching normal content, keep the output centered on:
- normal recognition
- normal review strategy
- variant recognition
- pitfall resistance

-----------------------------------------------------

FIGURE CAPTION RULE

Do not let a single figure caption dominate the extraction.

Image cards may use figures, but the card set must still reflect the full article when the workflow requires full-source review.

If captions are extracted as supplemental source text only:
- review them as a separate support layer
- mine them for supportable pearls
- do not let caption presence alone force image-based UNKNOWN generation

-----------------------------------------------------

COVERAGE-TO-CARD RULE

Checking a bucket does NOT mean forcing a card.

Generate a card only when the bucket contains:
- a distinct learning target
- enough specificity to support a clean card
- a fit within the allowed card types

If a bucket lacks enough support, omit the card rather than invent details.

-----------------------------------------------------

AUDIT FAILURE RULE

If the source was not reviewed in full, or if major supported buckets were ignored:
STOP.
Do not generate cards.

SECTION: SUMMARY RULE

SECTION PURPOSE

This file defines when the summary field is required and exactly how it must be written for the NORMAL engine.

-----------------------------------------------------

WHEN SUMMARY IS REQUIRED

The summary field MUST be populated for all cards derived from:
- a full RadPrimer article
- a full STATdx article

This applies regardless of card type:
- NORMAL UNKNOWN
- NORMAL DIFFERENTIAL DRILL
- NORMAL MECHANISM
- NORMAL BOARDS TRAP
- NORMAL HIGH-YIELD

If the card is not derived from a full RadPrimer or STATdx article:
- leave summary blank

-----------------------------------------------------

SUMMARY SOURCE DISCIPLINE

The summary must be derived ONLY from the full anchor article.

The summary must:
- reflect full-article review
- be article-level rather than card-level
- remain identical across all cards generated from the same article
- remain source-faithful
- stay inside the article's supported scope
- avoid outside knowledge
- avoid Core supplementation
- avoid Radiopaedia supplementation
- avoid unsupported inference

The summary is an auxiliary structured synthesis.
It is NOT the source of truth for card generation.
Cards must still come from the full article itself.

-----------------------------------------------------

SUMMARY FORMAT (STRICT)

The summary field must contain:
- no literal newline characters
- one continuous Anki-safe HTML string
- no raw markdown markers such as #, ##, **, *, -, or markdown tables

Use simple Anki-safe HTML only:
- <b>
- <br>
- <div>
- <ul>
- <ol>
- <li>
- light inline style attributes only when they improve hierarchy and remain robust in Anki

The summary must begin with:
- a visually distinct title block using the source article's actual title or main topic
- the title block should include a clean emoji or icon anchor
- then 1-2 sentences summarizing the main topic from the article only

SUMMARY ICON ENCODING RULE

For TSV export, do not write literal emoji characters directly into any field, including the summary field.

When the summary requires emoji or icon anchors, encode them as HTML numeric character references instead of literal Unicode characters.

Examples:
- Use `&#x1F9ED;` instead of the compass emoji
- Use `&#x1F50D;` instead of the magnifying glass emoji
- Use `&#x26A0;&#xFE0F;` instead of the warning emoji
- Use `&#x2B50;` instead of the star emoji

This prevents PowerShell, console, clipboard, or intermediate script encoding from converting icons into question marks during TSV generation.

The summary should still render with icons in Anki/HTML, but the TSV text itself should remain ASCII-safe.

The title block should:
- stand out visually from the rest of the summary
- use simple HTML hierarchy such as a bold title inside a <div>
- avoid hardcoded example entities
- allow light icon anchors only when they genuinely improve scanability, and encode them as HTML numeric character references

Then use clearly labeled sections only when supported by the article.

The summary should read like a polished radiology study handout:
- visually readable first
- easy to scan
- not plain or paragraph-heavy
- organized with clear spacing between major sections

Formatting hierarchy rules:
- major section headers must be on their own line
- subsection headers must also be on their own line
- explanatory content should appear underneath as bullets or nested bullets
- do NOT use inline `Header: long sentence` formatting for the main outline
- after the initial 1-2 sentence summary, prefer bullet-based and hierarchical formatting over prose blocks
- when choosing between fewer longer bullets versus more shorter nested bullets, prefer more shorter nested bullets

- <b>Summary</b>
- <b>Key Imaging Findings</b>
- <b>Differential Diagnosis</b>
- <b>Clinical Presentation</b>
- <b>Pearls / Pitfalls</b>
- <b>Key Measurements / Criteria / Scoring Systems</b>
- other article-supported labels when needed for readability in the NORMAL workflow, such as landmarks, structures, review approach, variants, discriminators, or modality anchors

Preferred section order when supported by the article:
- Abbreviations / Acronyms
- overview
- Key Imaging Findings
- Differential Diagnosis
- Clinical Presentation / Demographics
- Pearls / Pitfalls / Diagnostic Checklist
- Super Summary

Major sections should use tasteful emoji or icon anchors for readability.
Use varied, clean icons with restraint.
Every major section header should use a visible emoji or icon anchor, encoded for TSV export as an HTML numeric character reference rather than as a literal emoji character.
Do not use the same icon for every section unless necessary.

SECTION: EXAM APPROACH RULES
FILE:    2D_examapproachrules.txt
==============================================================================

SECTION PURPOSE

This file governs exam-approach educational content for the NORMAL engine.

Exam-approach content is allowed and important in this engine.
It must be mapped into the existing card types.
It does NOT justify creating new card types.

-----------------------------------------------------

EXAM-APPROACH CONTENT (ALLOWED)

This includes source-supported rules about:
- how to approach an ultrasound, CT, MRI, nuclear medicine, PET/SPECT, or scintigraphic exam
- what landmarks to identify first
- what planes matter
- what sequences matter
- what phases matter
- what tracer, uptake phase, delayed phase, excretion route, or biodistribution pattern matters
- what constitutes a complete normal review
- how to review structures in a safe order
- how to avoid overcalling disease in a normal study
- how to localize a less-visible structure from visible anchors
- what is truly seen versus mainly inferred on routine imaging

-----------------------------------------------------

ORIENTATION ANCHOR CARD RULE

For each normal-anatomy topic or subtopic, prioritize a small number of orientation-anchor cards that teach the learner what to identify first before attempting finer localization.

These cards should focus on practical visible anchors such as:
- ventricular landmarks
- major peduncles
- brainstem level
- major fissures
- cisterns
- hemisphere versus vermis
- large nuclei when actually visible

Do not let these become long exhaustive lists.
They should remain high-yield navigation cards.

-----------------------------------------------------

CARD-TYPE MAPPING RULE

Map exam-approach content into the existing allowed card types as follows:

NORMAL HIGH-YIELD
- compact approach rules
- landmark checklist
- normal workflow anchors
- review sequence
- complete-normal-review rules
- orientation-anchor cards
- topographic localization rules when the main task is conceptual rather than image identification

NORMAL MECHANISM
- why a plane, sequence, or phase is helpful
- why a structure changes appearance across modalities
- why a normal discriminator works
- why a localization anchor or sequence improves confidence when the source gives a real reason

NORMAL BOARDS TRAP
- classic approach errors
- omission traps
- pseudolesions created by incomplete review
- overcall-prone normal findings

NORMAL UNKNOWN
- image-based first-landmark identification
- recognition of the structure or relationship that anchors the review

NORMAL DIFFERENTIAL DRILL
- use only when the approach lesson depends on true discrimination
- examples: normal vs abnormal, variant vs pathology, mimic vs lesion

No new card type is required for exam-approach content.

-----------------------------------------------------

APPROACH CONTENT SHAPE RULE

Approach content must remain specific and cardable.

Good targets:
- first landmark to find
- order of review
- best phase to confirm a normal relationship
- key sequence for identifying a normal structure
- best visible anchors for localizing a less-visible structure
- direct-visibility versus inferred-localization teaching
- minimum checklist for a complete normal read
- feature that prevents overcalling pathology

Bad targets:
- long freeform protocol essays
- generic "systematic review is important" statements
- unsupported workflow advice

-----------------------------------------------------

OVERVIEW-TO-ANCHOR TRANSLATION RULE

When the source gives a broad anatomic overview, do not stop at a flat inventory of spaces, contents, or boundaries if that same material can be taught as a practical review path.

Broad cards should ideally tell the learner:
- what region or level to identify first
- which visible anchor structures organize the territory
- how adjacent compartments relate around those anchors
- what less-visible structure is then localized relative to them

When ultrasound is the source-highlighted modality, visible anchors may also include:
- echogenic landmark structures
- lumen-wall interfaces
- branching patterns
- probe plane or sweep direction
- near-field versus deep-field relationships

When nuclear medicine is the source-highlighted modality, anchors may include:
- exact tracer or exam
- expected physiologic uptake
- expected non-uptake or photopenia
- excretion pathway
- delayed or phase-specific distribution when source-supported
- planar versus SPECT/SPECT-CT versus PET/PET-CT localization context

If a broad card does none of that and only reproduces a taxonomy list, prefer rewriting it into an anchor-based review card or folding it into a stronger localization card.

-----------------------------------------------------

QUESTION SHAPING RULE

When both are source-supported, prefer prompts that train scan navigation over prompts that only train enumeration.

Prefer forms like:
- "Which landmarks should anchor a first-pass review of..."
- "How do you localize..."
- "What visible structures help place..."

Over forms like:
- "List all..."
- "Name all..."

unless the list itself is the compact high-yield anchor system being taught.

-----------------------------------------------------

REVERSE LOCALIZATION RULE

When source-supported, the engine may and should also teach anatomy in reverse-localization form.

That means cards may ask, in anatomy-first language:
- if a finding is in this compartment, what structure or region are you localizing to
- if the lesion abuts these landmarks, what space or level are you in
- which normal anchors tell you that you are in this cistern, fissure, compartment, or plane

These should remain normal-anatomy cards, not disease differentials.
Their purpose is to build the habit of localizing future abnormalities correctly.

-----------------------------------------------------

SYNTHESIS REVIEW CARD RULE

A small number of synthesis cards is encouraged when they materially improve reading fluency.

Good synthesis targets:
- the order to review a compact anatomic territory
- the anchor path across a region
- the minimal high-yield map of a compartment system

These cards must stay compact and operational.
Do not turn them into long catch-all summaries.

-----------------------------------------------------

MODALITY-SPECIFIC RULE

When the source provides modality-specific approach content, preserve the modality.

Examples of valid distinctions:
- ultrasound first-pass landmark logic
- CT phase-dependent review logic
- MRI sequence-dependent normal appearance logic
- nuclear medicine tracer-specific biodistribution and physiologic uptake logic
- PET/SPECT localization or attenuation-correction pitfall logic when source-supported

Do not flatten modality-specific advice into generic anatomy prose.

-----------------------------------------------------

OVERCALL-PREVENTION RULE

If the source teaches how to avoid overcalling disease in a normal study, preserve that teaching point.

Usually map it to:
- Boards_Trap / Boards_Trap_Q
OR
- Imaging_Differentiation
OR
- High_Yield_A when the teaching point is compact and checklist-style

-----------------------------------------------------

APPROACH SAFETY RULE

Do not create an exam-approach card unless the source supports a concrete actionable rule.
Do not invent review sequences, landmark priorities, or protocol preferences.

SECTION: ARTICLE DECOMPOSITION RULES
FILE:    2E_articledecompositionrules.txt
==============================================================================

SECTION PURPOSE

This file explains how to decompose a full RadPrimer or STATdx article into card-worthy content buckets for the NORMAL engine.

The goal is to extract the full educational value of a normal-anatomy article without defaulting to pathology-style output.

-----------------------------------------------------

ARTICLE DECOMPOSITION BUCKETS

For each full article, inspect whether it contains content for any of these buckets:

1) Named structures
- what structures must be recognized

2) Spatial relationships
- what normal adjacency, course, or orientation relationships matter

3) Landmark identifiers
- what landmarks anchor orientation or review

4) Modality appearance anchors
- what expected normal appearance is emphasized on ultrasound, CT, MRI, nuclear medicine, PET/SPECT, or radiograph

5) Exam approach steps
- what to look at first
- what order to review
- what must be checked to complete a normal read

6) Plane / sequence / phase logic
- what plane, sequence, or phase best demonstrates the normal concept

7) Measurement / morphology concepts
- only when explicitly supported by source

8) Normal variants
- what variant patterns are described as non-pathologic

9) Pseudolesions / mimics
- what normal findings may be mistaken for disease

10) Pitfalls
- where readers overcall or mislocalize the normal finding

11) Discriminators
- what feature separates normal from abnormal, or variant from pathology

12) Mechanistic explanations
- why the normal appearance looks the way it does
- why modality or phase matters

13) Landmark-based review strategy
- what sequence of landmarks or structures creates a reliable normal review

14) Radiology identification anchors
- what directly visible structures organize practical MRI, CT, or ultrasound localization
- what less-visible structures are placed relative to them

15) Ultrasound-specific anchors
- what echogenic structure, interface, lumen, wall, Doppler pattern, dynamic maneuver, probe plane, or sweep organizes normal recognition
- what ultrasound artifact, variant, or pseudolesion could be overcalled

16) Nuclear medicine-specific anchors
- what tracer or exam is being described
- what normal biodistribution, physiologic uptake, non-uptake, excretion, photopenia, timing, or pseudolesion pattern is emphasized
- what tracer-specific pitfall could be overcalled as disease

-----------------------------------------------------

DECOMPOSITION-TO-CARD-TYPE RULE

Map buckets into card types deliberately:

- named structures, landmarks, spatial relationships, and normal appearance anchors -> usually NORMAL UNKNOWN or NORMAL HIGH-YIELD
- exam approach steps, landmark-based review strategy, and radiology identification anchors -> usually NORMAL HIGH-YIELD
- plane / sequence / phase logic and mechanism-style explanations -> usually NORMAL MECHANISM
- variants, mimics, pitfalls, and overcall risks -> usually NORMAL BOARDS TRAP or NORMAL DIFFERENTIAL DRILL
- true discrimination between normal, variant, and pathology -> NORMAL DIFFERENTIAL DRILL
- ultrasound-specific anchors -> existing card types using 8B_ultrasoundappearance.txt
- nuclear medicine-specific anchors -> existing card types using 8C_nuclearmedicineappearance.txt

When a source paragraph is gross-anatomy heavy, first ask whether it is better expressed as:
- a visible-anchor card
- a localization card
- a review-order card

before defaulting to a list-style anatomy card.

-----------------------------------------------------

EXTRACTION MAP RULE

For a full article, create an extraction map before card generation when the article contains multiple meaningful buckets.

The extraction map should:
- list the supported buckets
- identify the best card family for each bucket
- order buckets according to the mastery progression rules
- make clear which buckets are foundational and which are advanced

The extraction map is especially important for staged generation.

-----------------------------------------------------

FOUNDATION-FIRST DECOMPOSITION RULE

Do not let decomposition jump straight to variants, traps, or mechanisms before the article's foundational normal anchors are secured.

In general:
- named structures, landmarks, relationships, and normal appearance anchors should be mapped first
- review strategy should be mapped before higher-order pitfalls
- variants, mimics, discriminators, and mechanisms should be layered on top of established foundations

-----------------------------------------------------

NO PATHOLOGY DEFAULT RULE

Do not let a normal-anatomy article drift into pathology-deck behavior.

If the article mentions abnormal entities only to teach separation, then:
- keep the card centered on the normal entity or discriminator
- limit abnormal mentions to what is required for the separator

-----------------------------------------------------

EXTRACTION GRANULARITY RULE

Split article content into distinct card-worthy units.

Do NOT compress the article into:
- one broad anatomy card
- one generic summary card
- one differential list

Also do NOT oversplit into trivial fragments.

Each card should test one stable, source-supported learning unit.

When possible, that unit should correspond to one usable imaging move rather than one isolated memorized fact.

-----------------------------------------------------

OVERVIEW CARD COMPRESSION RULE

Do not create multiple neighboring broad overview cards that repeat the same map of spaces with minor reshuffling.

When adjacent candidate cards substantially overlap, prefer:
- one stronger regional anchor card
- one orientation/review-order card
- one or more focused follow-up cards only when they add genuinely new localization utility

The goal is a stable anchor system, not a stack of near-duplicate list cards.

-----------------------------------------------------

CAPTION PEARL ADMISSION RULE

Caption-derived content must earn its place in the deck.

Prefer a caption-derived card only when the caption materially improves one of the following:
- practical localization
- visible-anchor recognition
- modality, sequence, phase, or probe-plane understanding
- vascular or compartmental orientation
- overcall prevention
- normal-variant honesty

If a caption only restates a low-yield fact without improving identification skill, fold it into a stronger card or omit it.

-----------------------------------------------------

MODEST CARD COUNT, HIGHER DENSITY RULE

Prefer fewer stronger cards over a larger number of thin or overlapping cards.

It is acceptable to generate fewer cards if doing so improves:
- identification skill
- localization clarity
- review-path teaching
- nonredundancy
- long-term usefulness in practice

-----------------------------------------------------

MATERIAL-TO-IMAGING TRANSLATION RULE

If the article teaches boundaries, contents, or compartment names, convert that material into imaging-useful form whenever possible.

Good translation moves include:
- organizing the answer around visible anchor structures
- stating which structures are directly appreciable
- stating which structures are mainly localized topographically
- linking the compartment to a practical modality-specific review move

For ultrasound-heavy topics, this includes:
- echogenic anchor recognition
- lumen or wall interface recognition
- branching-pattern localization
- probe-plane and sweep-based orientation

For nuclear medicine-heavy topics, this includes:
- tracer-specific normal biodistribution
- physiologic uptake and excretion pathways
- timing or phase when source-supported
- normal variants and pseudolesions that alter uptake or apparent activity
- the discriminator that prevents mistaking physiologic activity for disease

Do not reduce radiology anatomy to textbook recitation when the source supports a more operational framing.

-----------------------------------------------------

DECOMPOSITION SAFETY RULE

If a bucket is unsupported, leave it empty.
If a supported bucket does not fit any allowed card type, omit it rather than inventing a new trigger.

If you believe a future new card type could help, leave a clearly labeled recommendation comment outside the card-generation workflow.
Do NOT implement a new card type.

SECTION: MASTERY PROGRESSION RULES
FILE:    2F_masteryprogressionrules.txt
==============================================================================

SECTION PURPOSE

This file defines the pedagogic order for extracting and generating NORMAL-engine cards from a full topic.

The goal is topic mastery, not merely card count.

-----------------------------------------------------

MASTERY PROGRESSION RULE

When extracting a full article, prioritize supported content in this order:

1) foundational structures and landmarks
2) orientation anchors and directly visible imaging anchors
3) normal spatial relationships
4) modality-specific normal appearance anchors
5) topographic localization rules for less-visible structures
6) exam-approach and review-sequence rules
7) normal variants and pseudolesions
8) discriminators between normal, variant, and abnormal
9) mechanism-style explanations

Within the foundational half of the progression, prefer cards that help the learner identify structures on the scan over cards that merely restate an anatomic list.

This order is mandatory unless the source contains only a narrow subset.

-----------------------------------------------------

FOUNDATION-FIRST RULE

Do not overproduce:
- variants
- traps
- mimics
- mechanisms

before the foundational normal anchors are established.

If the article supports foundational anchors, they must be encoded before advanced or failure-mode cards dominate the output.

-----------------------------------------------------

MASTERY BALANCE RULE

For long or multi-bucket topics:
- ensure the card set contains enough foundational anatomy and normal-appearance anchors to support later trap or discriminator cards
- do not let pitfall-heavy output outnumber the anchors needed to understand those pitfalls
- do not let variant cards dominate a topic whose core normal relationships have not yet been encoded

-----------------------------------------------------

PROGRESSION-TO-CARD-FAMILY RULE

Typical progression by card family:
- foundational structures, landmarks, directly visible anchors, and normal appearance anchors -> NORMAL UNKNOWN or NORMAL HIGH-YIELD
- orientation anchors, topographic localization rules, and review sequence anchors -> NORMAL HIGH-YIELD
- variants and mimics -> NORMAL BOARDS TRAP or NORMAL HIGH-YIELD
- true separators -> NORMAL DIFFERENTIAL DRILL
- why a normal pattern or discriminator works -> NORMAL MECHANISM

If two candidate foundational cards cover nearly the same regional map, prefer the one with greater scan-navigation utility.

Allow a small number of compact synthesis review cards when they strengthen regional reading fluency and unify multiple foundational anchors into one usable review path.

This is a prioritization rule, not a new trigger system.

-----------------------------------------------------

MASTERY AUDIT RULE

Before final export for a full topic, confirm:
- the foundational buckets were not skipped
- the topic has enough anatomy and normal-review anchors to support higher-order cards
- advanced cards are layered on top of established normal understanding

If the output is trap-heavy or variant-heavy without adequate foundations:
- rebalance before export

SECTION: ADAPTIVE GENERATION RULES
FILE:    2G_adaptivegenerationrules.txt
==============================================================================

SECTION PURPOSE

This file defines when the NORMAL engine may use one-shot generation and when it should switch to staged generation.

Quality and coverage take priority over speed.

-----------------------------------------------------

ONE-SHOT GENERATION RULE

One-shot generation is allowed when the article is:
- short
- focused on one narrow concept cluster
- low complexity
- supported by a small number of clean buckets
- unlikely to lose pedagogic sequence if generated in one pass

One-shot generation is a permission, not a default mandate.

-----------------------------------------------------

STAGED GENERATION RULE

Use staged generation when the article is:
- long
- multi-bucket
- multi-modality
- image-heavy
- rich in variants, pitfalls, or discriminators
- likely to suffer from coverage loss or bad sequencing if generated in one pass

When quality would drop in one-shot mode, staged generation is preferred.

-----------------------------------------------------

PREFERRED STAGED WORKFLOW

For staged generation, use this order:

1) full-source coverage audit
2) extraction map or bucket map
3) generate cards by bucket, section, or card family
4) perform a final structural and pedagogic audit
5) export only after the audit passes

-----------------------------------------------------

ADAPTIVE DECISION RULE

Before generating cards from a full article, decide intentionally:
- one-shot
OR
- staged

Base the decision on:
- article length
- bucket count
- conceptual breadth
- number of supported images
- number of supported variants, mimics, or traps
- risk of losing mastery progression in a one-pass output

Do not blindly generate everything in one shot for long articles.

-----------------------------------------------------

STAGED-GENERATION CONTENT RULE

When staged generation is used:
- generate foundational mastery anchors before advanced buckets
- keep bucket boundaries clear
- avoid mixing every supported card family into the first pass
- defer final export until the structural and pedagogic audit is complete

-----------------------------------------------------

SMALL-ARTICLE EFFICIENCY RULE

Do not force staging for every tiny article.

If the article is small and low complexity:
- one-shot generation is acceptable
- coverage audit and structural rules still apply

- Mechanism_Q
- Boards_Trap_Q
- High_Yield_Q

If more than one is populated:
- the note is structurally invalid
- split into separate notes

TRIGGER SAFETY ENFORCEMENT

If Image is populated but Question is empty:
- the card must NOT be treated as NORMAL UNKNOWN

Populate only the fields required for the card type you are intentionally creating.
No unintended card types may be generated.

-----------------------------------------------------

DUAL-TRIGGER PROHIBITION

The following trigger combinations are forbidden on the same note:
- Question plus Differentials
- Question plus Mechanism
- Question plus Boards_Trap
- Question plus High_Yield_Q
- Differentials plus Mechanism
- Differentials plus Boards_Trap
- Differentials plus High_Yield_Q

- Boards_Trap plus High_Yield_Q

- clear the unintended field(s) or split into separate notes

FIELD POPULATION DISCIPLINE

The existence of a field in the shared note type does NOT justify populating it.

Never fill a field:
- just because it exists
- to make a card feel more complete
- to mirror another card type
- to avoid a blank

Only populate a field when the intended card type explicitly allows it.

-----------------------------------------------------

FRONT-SIDE BLINDING RULE

Fronts must contain only the elements required by their specific template.

No front may include:
- answer leakage
- named diagnoses unless the card intentionally requires a tightly bounded mimic-rejection anchor
- caption-derived answer statements
- management clues
- interpretive hints that collapse the task

All fronts must preserve:
- anatomy recognition
- normal-pattern recognition
- normal-vs-variant discrimination
- pitfall resistance

-----------------------------------------------------

GLOBAL NO-METADATA CARD RULE

Cards must not ask about prompt workflow, source review, extraction audit, what the article contained, what the source supported encoding, or what a generator decided to include.

Do not export fronts such as:
- "What did the source review support encoding?"
- "What information was included in this article?"
- "What does the prompt require?"
- "What did extraction identify?"
- "What should be encoded from this source?"

Convert those drafts into direct anatomy, landmark, normal-appearance, review-sequence, mechanism, or pitfall questions. If no direct learning target remains, omit the card.

Back fields may include source-supported teaching content, but they must not read like generation metadata or internal audit notes.

-----------------------------------------------------

GLOBAL INTERPRETIVE RULE

These card types exist to train NORMAL material, not pathology-first diagnosis.

In this engine:

NORMAL UNKNOWN
- identify a normal structure, landmark, relationship, expected normal appearance, physiologic appearance, or normal variant from image review

NORMAL DIFFERENTIAL DRILL
- distinguish normal vs abnormal, variant vs pathology, or mimic vs lesion using the dominant discriminator

NORMAL MECHANISM
- explain the anatomic, developmental, physiologic, modality, sequence, or phase-based reason a normal appearance or discriminator works

NORMAL BOARDS TRAP
- encode classic normal mimics, pseudolesions, overcalls, omission traps, and normal variants commonly mistaken for pathology

NORMAL HIGH-YIELD
- encode compact anatomy definitions, landmark rules, approach rules, modality anchors, checklist items, normal measurements when source-supported, and variant anchors

-----------------------------------------------------

STRUCTURAL TEACHING PRIORITY RULE

If the source is organized around a structure that must be followed, the deck must first teach that structure as a usable map before relying on isolated detail cards.

Examples include:
- ordered review sequence
- vessel, duct, nerve, or tract course
- layer stack
- compartment or space boundaries
- branch, segment, or level map
- visible landmark to inferred anatomy pathway
- normal variant versus pseudolesion decision path
- modality-specific normal exam approach

Preferred mappings:
- the framework itself -> NORMAL HIGH-YIELD
- why the framework works -> NORMAL MECHANISM
- visible anchor landmark -> NORMAL UNKNOWN or NORMAL HIGH-YIELD
- common overcall from breaking the framework -> NORMAL BOARDS TRAP
- normal-vs-abnormal discriminator -> NORMAL DIFFERENTIAL DRILL

When order matters, the answer must preserve the order. Do not scramble source-stated sequences into flat unordered lists.

-----------------------------------------------------

VISIBILITY-FIRST CANDIDATE EVALUATION RULE

For each candidate anatomy entity, ask in order:
- Is this structure a true high-yield imaging anchor?
- Is it directly visible, variably visible, or mainly inferred?
- If not directly visible, what visible structures best localize it?
- Is it worth a dedicated card, or should it be folded into a broader orientation/localization card to avoid deck inflation?

Do not create a dedicated note just because a named structure appears once in the text.

-----------------------------------------------------

ATOMIC RETRIEVAL UNIT RULE

Each card should test one primary retrieval target.

Do not combine multiple independent retrieval tasks into one card.

Common overpacked combinations that usually require splitting:
- boundaries/location of a space PLUS contents of that space
- localization rule PLUS full structure inventory
- modality appearance PLUS variant/pitfall unless the pitfall is the reason the appearance matters
- definition PLUS clinical relevance PLUS review sequence
- multiple adjacent compartments that each require separate landmarks

Allowed multi-fact answers are limited to facts that directly support the same retrieval target.

Multiple boundaries or adjacent landmarks are allowed in one card when they all serve the same localization target.
Do not split a boundary-rich localization card merely because it uses several boundaries to place one structure or space.

In general:
- one primary answer
- up to 2-3 supporting anchors
- one compact clinical or overcall relevance line when source-supported

If the answer needs more than 3-4 distinct facts, split it unless it is a rare intentional framework, pathway, sequence, or synthesis card.

Framework/pathway exceptions are allowed only when the single retrieval target is the structure of the map itself, such as:
- an ordered review path
- a vessel, duct, nerve, or tract course
- a layer stack
- a compartment map
- a normal variant decision pathway

Even then, the answer must stay organized, ordered when order matters, and compact enough for active recall.

Long integrative teaching belongs in the narrative, not in routine Anki backs.

-----------------------------------------------------

EXAM-APPROACH MAPPING RULE

Exam-approach content must stay inside the existing card types.

Preferred mappings:
- review sequence, landmark checklist, completeness rules -> NORMAL HIGH-YIELD
- why a phase, sequence, or plane helps -> NORMAL MECHANISM
- common overcall or omission errors -> NORMAL BOARDS TRAP
- image-based anchor landmark identification -> NORMAL UNKNOWN
- true discriminator tasks -> NORMAL DIFFERENTIAL DRILL

Do not invent a new approach card type.

-----------------------------------------------------

CARD-TYPE SELECTION FOR LOCALIZATION RULE

Preferred mappings for localization-focused teaching are:
- directly visible high-yield anchors -> NORMAL UNKNOWN or NORMAL HIGH-YIELD
- variably visible structures with stable landmark logic -> NORMAL HIGH-YIELD, sometimes NORMAL UNKNOWN when the image truly supports recognition
- mainly topographically inferred structures -> NORMAL HIGH-YIELD, or NORMAL BOARDS TRAP when mislocalization creates a recurrent overcall
- causal explanation of why a localization rule or sequence works -> NORMAL MECHANISM

If the structure is mainly topographically inferred, do not default to a pure image-identification UNKNOWN card.

-----------------------------------------------------

ARCHITECTURE SAFETY RULE

The same field can only mean one trigger at a time.

Therefore:
- never populate Differentials unless intentionally generating NORMAL DIFFERENTIAL DRILL
- never populate Mechanism unless intentionally generating NORMAL MECHANISM
- never populate Boards_Trap unless intentionally generating NORMAL BOARDS TRAP
- never populate High_Yield_Q unless intentionally generating NORMAL HIGH-YIELD
- never populate Question for a non-NORMAL-UNKNOWN image card

This rule is absolute.

SECTION: IMAGE RULES
FILE:    imagerules.txt
==============================================================================

SECTION PURPOSE

This file governs image usage, image dependency, and image-question safety for the NORMAL engine.

-----------------------------------------------------

IMAGE SELECTION RULE

Use images only when the image materially improves the learning task.

Appropriate uses:
- structure recognition
- landmark recognition
- spatial relationship recognition
- modality-specific normal appearance
- variant recognition
- pseudolesion or mimic recognition
- sequence or phase comparison when source-supported

Do not add an image just to imitate an UNKNOWN card if the learning point is purely conceptual.

If the main teaching point is topographic localization of a structure that is not reliably resolved on routine imaging:
- prefer a localization card over a fake crisp-identification image card
- use the image only if it genuinely helps anchor the visible landmarks used for localization

-----------------------------------------------------

IMAGE IDENTIFICATION SAFETY RULE

Do not generate image-identification cards that ask the learner to identify a structure as though it were clearly visible if the structure is usually topographically inferred.

Instead:
- convert those to localization cards
- or phrase them as "Which visible landmarks help localize X?"
- or "Where would X lie relative to Y and Z on MRI/CT?"

This is especially important for small nuclei, tracts, and fine subdivisions that are often not discretely resolved on routine clinical studies.

-----------------------------------------------------

CAPTION PEARL EXTRACTION RULE

Captions must be reviewed independently from NORMAL UNKNOWN eligibility.

Useful captions may supply:
- landmark pearls
- localization clues
- sequence, phase, or plane pearls
- vascular territory clues
- pseudolesion distinctions
- pitfall prevention
- visibility-status clarifications
- structure-to-structure relationships

Do not discard an informative caption merely because it does not justify an image-based UNKNOWN card.

-----------------------------------------------------

CAPTION PEARL MUST EARN THE CARD RULE

An informative caption does not automatically deserve its own card.

Before promoting caption content into a separate card, ask:
- does this improve structure identification
- does this improve localization or review order
- does this prevent an overcall
- does this clarify visibility honesty
- does this add a modality-specific pearl the article text did not already capture well

If not, absorb it into a stronger existing card or omit it.

-----------------------------------------------------

IMAGE UTILITY TRIAGE RULE

For each image-caption pair, classify it separately as:
- UNKNOWN-worthy image
- support-only image
- non-contributory image

A support-only image/caption pair may still feed non-UNKNOWN cards and explanatory fields even if it does not justify an image-based UNKNOWN card.

-----------------------------------------------------

IMAGE-OPTIONAL ARTICLE RULE

The NORMAL engine does not require images for every article.

Therefore:
- image-based extraction is optional
- NORMAL UNKNOWN cards are optional
- Section A or image-first extraction may be absent
- the absence of usable images is not a workflow failure

If an article has no usable source-supported images:
- do not fabricate Image
- do not fabricate Image_Annotated
- do not fabricate Question
- do not fabricate UNKNOWN cards
- proceed with concept-driven card types only when supported

If the workflow explicitly states that no pictures are being provided for the run:
- treat that as a deliberate no-image workflow
- do not create NORMAL UNKNOWN cards
- do not ask for or expect image recovery inside the run
- proceed with non-image card families only

-----------------------------------------------------

CAPTIONS-ONLY WORKFLOW RULE

When the workflow context says captions are being extracted as supplemental source text only:
- do not expect image files
- do not create image-based UNKNOWN cards by default
- mine captions for supportable pearls
- route useful caption content into existing card families such as NORMAL HIGH-YIELD, NORMAL BOARDS TRAP, NORMAL MECHANISM, NORMAL DIFFERENTIAL DRILL, and explanatory back-side fields
- do not treat caption presence alone as justification for an image card

-----------------------------------------------------

IMAGE / QUESTION SAFETY RULE

Question may be populated only for NORMAL UNKNOWN.

Therefore:
- NORMAL UNKNOWN requires Image and Question
- any non-NORMAL-UNKNOWN card may include Image only if Question remains blank
- Image must never be used to justify populating Question on a non-UNKNOWN card

If Image is present and a non-UNKNOWN trigger field is populated:
- Question must remain blank

If Image, Question, and another trigger field coexist:
- the note is structurally invalid

-----------------------------------------------------

IMAGE DEPENDENCY RULES

Image_Annotated must not be populated unless Image is also populated.

Original_Caption must remain blank unless Image is also populated.

Annotated-only image notes are not supported by this engine.

Image support fields must not create hidden card-family contamination.

Therefore:
- Image_Annotated does not authorize a note by itself
- Original_Caption does not authorize a note by itself
- image support fields must remain subordinate to the intended card type

-----------------------------------------------------

IMAGE SOURCE DISCIPLINE

Images must remain compatible with the shared media pipeline.

Do not break:
- shared media references
- filename handling
- image ordering
- import compatibility

If multiple images are used:
- preserve source order exactly
- preserve annotated and non-annotated pairing exactly

MULTI-IMAGE TEMPORAL / PROCEDURAL CONTEXT-LABEL RULE

If a single card uses multiple images and the captions clearly indicate that the images are not all from the same timepoint, phase, status, view state, or procedural state, the image stack must make that relationship explicit before the relevant image is shown.

Caption-supported context examples include:
- baseline / initial study
- follow-up or later study, including a stated interval
- preoperative / postoperative
- pre-procedure / post-procedure
- before treatment / after treatment
- arterial phase / portal venous phase / delayed phase
- precontrast / postcontrast
- neutral normal-comparison or variant-comparison image
- same subject after intervention, progression, resolution, or expected postoperative/treatment change

Rules:
- Add a brief neutral context label immediately before each affected image.
- The label must be derived only from the source caption or explicit case grouping.
- The label must clarify time, phase, sequence, view state, or procedural status.
- The label must not name a hidden structure, diagnosis, named sign, answer, differential, or caption-derived visual clue that would solve the card on the front.
- Do not add context labels to single-image cards.
- Do not add context labels to multi-image stacks when all images are clearly same-time, same-status, or same-sequence without meaningful time/procedure/phase/view-state change.
- Do not modify the verbatim caption in Image_Annotated stackCap or Original_Caption.
- This rule is a narrow exception to any usual front-side plain-image/no-wrapper rule.

Allowed front-side Image format when the exception is needed:
<div class="stackItem"><div class="stackContext">Initial image.</div><img src="FILE1.ext"></div><br><div class="stackItem"><div class="stackContext">Follow-up image 2 weeks later.</div><img src="FILE2.ext"></div>

Allowed back-side Image_Annotated format when the exception is needed:
<div class="stackItem"><div class="stackContext">Postoperative image.</div><img src="FILE2_annot.ext"><div class="stackCap">VERBATIM CAPTION</div></div>

If the extracted input labels image blocks as `CASE_##`:
- treat each provided case block as an explicit input unit
- account for every provided case block during generation
- do not silently omit a provided case block

Before export, every selected image or explicit CASE block must be accounted for as:
- carded as an image-recognition/localization/normal-appearance card
- used as support inside another source-supported card
- intentionally omitted as redundant, non-contributory, or not genuinely visible

Do not silently skip selected teaching graphics, normal comparators, modality comparators, pitfall images, pseudolesion examples, or workflow diagrams. If they are not UNKNOWN-worthy, route their useful teaching point into a high-yield, boards-trap, mechanism, localization, or differential-discriminator card when source-supported.

-----------------------------------------------------

ANNOTATION RULE

Annotated images are allowed when labels materially improve learning.

Use annotated images for:
- named structures
- landmarks
- spatial relationships
- localization of a variant or mimic

Do NOT add unsupported labels or interpretations.
Do NOT annotate beyond what the source or approved workflow supports.

-----------------------------------------------------

ANSWER-LEAKAGE RULE

Images must not leak the answer through:
- visible answer text embedded by the generator
- unsupported callouts
- revealing captions placed on the front
- front-side wording that names the target directly

The image should support recognition, not replace it.

-----------------------------------------------------

CAPTION RULE

If Original_Caption is required by the workflow:
- keep it verbatim
- do not paraphrase
- do not convert it into interpretive teaching text

If Original_Caption is not required:
- leave it blank

-----------------------------------------------------

MODALITY CONTEXT RULE

If the learning target depends on modality, sequence, plane, or phase:
- preserve that context in the image card design
- do not flatten MRI sequence-specific or CT phase-specific teaching into a generic image prompt

If the image comes from an annotated teaching figure or unusually high-resolution depiction:
- do not let that alone justify claiming routine clinical visibility
- preserve modality honesty in the associated teaching fields

-----------------------------------------------------

IMAGE SAFETY RULE

If the available image does not cleanly support the target learning point:
- do not force an image card
- use a concept card instead

IMPLEMENTATION NOTE

No new trigger logic is implemented here.
If a future dedicated Radiopaedia drill is desired, it must be added intentionally without breaking the existing shared chassis.


==============================================================================
SECTION: NORMAL TAG RULES
FILE:    tagrules_normal.txt
==============================================================================

SECTION PURPOSE

This file defines the NORMAL engine's tag architecture at the policy level.

Tags remain part of the shared TSV/export architecture, but in this workflow they are generated automatically downstream.

Therefore this file defines:
- tag axes
- tag expectations
- compatibility rules

It does NOT require verbose manual tag writing during normal card-content generation unless a workflow explicitly asks for it.

-----------------------------------------------------

SHARED TAG ARCHITECTURE RULE

Keep the existing shared tagging architecture.
Do not invent a separate incompatible NORMAL-only schema.

NORMAL-engine tag logic must remain derivable inside the shared downstream tagging pipeline.

-----------------------------------------------------

CONCEPTUAL TAG AXES

The NORMAL engine should conceptually support tags across these axes:
- Engine
- CardType
- Source
- EntityType
- Modality
- Region or System
- Yield
- optional distinctions such as Variant, Mimic, Approach, or Pitfall when supported by the wider shared system

-----------------------------------------------------

TAG DERIVATION RULE

Downstream automatic tagging may derive tags from:
- the active engine
- the triggered card type
- source provenance
- entity definition
- modality context
- region or system context
- yield or board-relevance level
- supported distinctions such as variant, mimic, approach, or pitfall

Tags should be derivable from card structure and supported content.
Do not fabricate unsupported tags.

-----------------------------------------------------

MANUAL TAGGING RULE

Do not burden routine card-content generation with manual per-card tag composition unless the workflow explicitly requires it.

The content generator should focus on:
- correct card structure
- correct field usage
- correct source support
- correct pedagogic sequencing

Tag population may occur automatically downstream.

-----------------------------------------------------

REQUIRED VS OPTIONAL TAG EXPECTATIONS

Conceptually required axes:
- Engine
- CardType
- Source when known

Conceptually preferred axes when supported by the content or workflow:
- EntityType
- Modality
- Region or System
- Yield

Optional axes only when clearly supported and recognized by the wider shared system:
- Variant
- Mimic
- Approach
- Pitfall

-----------------------------------------------------

TAG SAFETY RULE

Tags must remain supportable by the actual card content and source.

Do not:
- fabricate region tags not supported by the topic
- fabricate modality tags for concept-only cards with no modality anchor
- fabricate variant or pitfall distinctions when the card does not actually encode them

The tag policy should help downstream automation, not distort the card set.

SECTION PURPOSE

This file defines output compatibility and pre-export structural validation for the NORMAL engine.

OUTPUT SHAPE RULE

The engine must output records compatible with the shared note type and TSV workflow.

Each TSV row must contain:
- the 21 note fields in the established order
- one Tags column at the end

Total columns: 22

Do NOT reorder columns.
Do NOT add columns.
Do NOT remove columns.

3. Image_Annotated
4. Question
5. Most_Likely_Diagnosis
6. Entity_Label
7. Differential_Q

9. Imaging_Differentiation

20. Radiopaedia_Case_Differential

-----------------------------------------------------

VALUE FORMATTING RULE

Output values must preserve shared import compatibility.

- do not insert extra columns
- do not emit placeholder tokens such as N/A
- use blank fields when content is intentionally absent
- keep per-field formatting controlled and deterministic
- do not use literal newline characters inside TSV fields
- use <br> where internal formatting is required

For summary specifically:
- use simple Anki-safe HTML only
- no literal newline characters
- do not emit raw markdown syntax
- keep the same article-level summary content across all rows derived from the same article

For Tags specifically:
- keep the final Tags column present in every TSV row
- export the final Tags column as blank / empty for every row

-----------------------------------------------------

PRE-TSV VALIDATION GATE

Before export, all of the following must be true.

A) Trigger integrity
- NORMAL UNKNOWN requires Image and Question
- if Image is populated but Question is blank, the note must not be treated as NORMAL UNKNOWN
- any non-NORMAL-UNKNOWN note with Image must keep Question blank
- NORMAL DIFFERENTIAL DRILL exists only when Differentials is populated
- NORMAL MECHANISM exists only when Mechanism is populated
- NORMAL BOARDS TRAP exists only when Boards_Trap is populated
- NORMAL HIGH-YIELD exists only when High_Yield_Q is populated
- no dual-trigger note may be exported

B) Forward dependency integrity
- if Differentials is populated, Differential_Q and Imaging_Differentiation must be populated
- if Mechanism is populated, Mechanism_Q and Entity_Label must be populated
- if Boards_Trap is populated, Boards_Trap_Q and Entity_Label must be populated
- if High_Yield_Q is populated, High_Yield_A must be populated

C) Reverse dependency integrity
- if Differential_Q is populated, Differentials must also be populated
- if Mechanism_Q is populated, Mechanism must also be populated
- if Boards_Trap_Q is populated, Boards_Trap must also be populated
- if High_Yield_A is populated, High_Yield_Q must also be populated
- if Image_Annotated is populated, Image must also be populated
- if Original_Caption is populated, Image must also be populated
- orphan companion fields are structural failures

D) Single-front-family integrity
- only one of the following may be populated on a note:
  Question
  Differential_Q
  Mechanism_Q
  Boards_Trap_Q
  High_Yield_Q
- if more than one is populated, split the content into separate notes

E) Forbidden-field integrity
- UNKNOWN must leave all non-UNKNOWN trigger fields and front-driving prompt families blank
- DIFFERENTIAL DRILL must leave Question, Mechanism_Q, Mechanism, Boards_Trap_Q, Boards_Trap, High_Yield_Q, and High_Yield_A blank
- MECHANISM must leave Question, Most_Likely_Diagnosis, Differential_Q, Differentials, Boards_Trap_Q, Boards_Trap, High_Yield_Q, and High_Yield_A blank
- BOARDS TRAP must leave Question, Differential_Q, Differentials, Mechanism_Q, Mechanism, High_Yield_Q, and High_Yield_A blank
- HIGH-YIELD must leave Question, Most_Likely_Diagnosis, Differential_Q, Differentials, Mechanism_Q, Mechanism, Boards_Trap_Q, and Boards_Trap blank

F) Image dependency integrity
- annotated-only image notes are not supported
- Image_Annotated must remain subordinate to Image
- Original_Caption must remain subordinate to Image
- image support fields must not create hidden prompt-family contamination
- if a multi-image card has caption-supported time/procedure/phase/view-state differences, neutral context labels must appear before the affected images, must not leak the answer, and must not alter verbatim captions

G) Source and summary integrity
- whole-source review requirements must be satisfied before card generation
- summary must never substitute for full-source review
- if a card originates from RadPrimer or STATdx, summary must be populated
- if a card does not originate from RadPrimer or STATdx, summary must be blank
- summary must remain article-only with no Core mixing, no outside facts, and no unsupported inference

G2) Image-optional article integrity
- the absence of usable images is not a structural failure
- zero NORMAL UNKNOWN cards is valid when no usable source-supported images exist
- do not force Section A or image-first extraction when the article has no usable images
- do not fabricate Image, Image_Annotated, or Question to create image-based cards
- concept-driven card families may export without any image-based notes when source-supported
- if the workflow explicitly states that no pictures are being provided for the run, zero NORMAL UNKNOWN cards is the expected outcome
- in that no-picture workflow, do not treat the absence of UNKNOWN cards as a coverage failure

G2C) Captions-only workflow integrity
- if the workflow context says captions are being extracted as supplemental source text only, zero NORMAL UNKNOWN cards is acceptable
- captions-only mode is not a workflow failure
- do not fabricate Image, Image_Annotated, or Question fields just because captions were present
- informative captions should still be mined for supportable pearls and routed into existing non-UNKNOWN card families or explanatory fields

G2B) Case-accounting integrity for case-labeled input
- if the extracted input presents image blocks as `CASE_##`, every provided case block must be accounted for
- no provided case block may be silently omitted
- if a case cannot support the expected image-driven card family, the output must make that explicit rather than dropping it silently

G3) Mastery progression integrity
- for full-topic extraction, foundational anatomy and normal-review anchors must not be skipped in favor of variants, traps, or mechanisms
- advanced cards must sit on top of established foundational anchors when the source supports those foundations
- if the final set is trap-heavy or variant-heavy without adequate anchors, rebalance before export

G4) Adaptive generation integrity
- do not assume one-shot generation for every article
- if the article is long, multi-bucket, or otherwise high complexity, use staged generation or staged internal planning before export
- one-shot generation is acceptable for short, focused, low-complexity articles
- quality and coverage outrank speed

G5) Visibility and localization integrity
- do not export a card that falsely implies a mainly topographically inferred structure is crisply and directly visible on routine imaging
- if the target is mainly inferred, convert the teaching into a localization-oriented card or broaden it into an orientation-anchor card
- localization teaching must live inside existing explanatory fields only; do not add schema columns
- where relevant, explanatory back-side content should distinguish directly visible versus variably visible versus mainly inferred
- cards should prefer stable landmark systems over low-yield micro-localization inflation

G5B) Radiology-identification integrity
- for a normal-anatomy topic, the deck must materially improve the learner's ability to identify the structures of interest on the source-highlighted modality, including MRI, CT, or ultrasound when applicable
- if a card is broad or overview-style, it should still teach a practical anchor system, review path, or localization rule rather than only a taxonomy list
- if neighboring overview cards substantially overlap, compress or rewrite them into fewer stronger anchor-based cards before export
- prefer fewer stronger cards over more thin cards when the tradeoff improves identification skill
- caption-derived standalone cards should survive only when they materially add localization, modality understanding, visibility honesty, or overcall prevention
- each retained card should ideally teach one stable imaging map move rather than a loose bundle of facts
- when source-supported, the deck should also preserve compact clinical usefulness such as lesion localization value, stroke-localization value, procedural relevance, or overcall prevention
- reverse-localization cards are encouraged when they improve practical reading skill without drifting into pathology-deck behavior
- comparison cards should remain only when they sharpen interpretation rather than merely increasing card count
- synthesis cards should remain few and operational, not summary-like or bloated
- standalone trivia that does not improve reading skill or judgment should be omitted

G5C) Atomic retrieval integrity
- each card must have one primary retrieval target
- routine cards should not combine boundaries/location, contents, imaging localization, modality appearance, pitfall, and clinical relevance into one overloaded answer
- boundaries/location of a space and contents of that space are usually separate retrieval targets
- multiple boundaries or adjacent landmarks are acceptable when they all support one localization target
- localization method and full structure inventory are usually separate retrieval targets
- if a card answer contains more than 3-4 distinct facts, audit it for splitting
- synthesis, framework, pathway, and ordered-sequence cards are the only routine exceptions, and they must be explicitly compact, intentional, and few
- a framework/pathway exception is valid only when the one tested concept is the map itself, not a bundle of unrelated facts
- if a card is overloaded, split it into separate cards or rewrite it around one primary map move before export

G5D) Normal structure/framework integrity
- if the source teaches an ordered review path, layer stack, vessel/duct/nerve course, tract course, compartment/boundary map, branch/segment map, or modality workflow, at least one card must encode that structure as a usable framework
- when order matters, the answer must preserve the order
- if the deck contains many isolated anatomy facts from a structured topic but no map, sequence, or framework card, repair the deck before export
- if a framework is too large, split it by region, level, course, or compartment rather than flattening it into trivia
- framework cards should support practical image review, localization, or normal-variant discrimination

G5E) Metadata-card ban
- do not export cards whose main target is source-review bookkeeping, prompt metadata, extraction audit, or generator decisions
- banned front shapes include "what did the source review support encoding", "what was included", "what does the prompt require", "what information was extracted", and similar variants
- banned back shapes include statements whose main function is "the source did/did not include this metric" unless the actual tested target is a direct anatomy or imaging fact
- convert metadata-shaped drafts into direct anatomy, landmark, review-path, normal-appearance, mechanism, or pitfall questions
- if conversion leaves no useful retrieval target, omit the card

G5F) External mechanism/development provenance integrity
- outside literature may appear only in mechanism/development/physiology/modality-physics clarification, stable quantitative facts, stable performance facts, or the approved localization overlay
- outside literature must clarify an article-supported fact; it must not introduce a new unsupported card target
- any externally supported explanation must include the exact label "External literature source (not stated in RadPrimer/STATdx/Core/Radiopaedia):" and an exact source link in the same field
- article-supported facts and externally clarified explanations must remain distinguishable
- summary must remain article-only and must not include outside literature content

G5G) Normal mimic and disease-mimic integrity
- if the source names disease mimics or pathology overcalls, encode them when they materially improve normal-vs-abnormal discrimination or pitfall resistance
- route true separator tasks to NORMAL DIFFERENTIAL DRILL
- route overcall-prevention/pseudolesion traps to NORMAL BOARDS TRAP
- do not inflate into broad pathology differential lists
- the card must remain anchored to normal anatomy, normal appearance, variant recognition, or pseudolesion recognition

G5H) Ultrasound-specific integrity
- if the source supports ultrasound-specific normal appearance, probe approach, Doppler, dynamic maneuver, artifact, pseudolesion, or discriminator content, encode it using the existing card families and the ultrasound module
- ultrasound cards must preserve the source-stated probe plane, sweep, echogenicity, interface, Doppler, acoustic, artifact, or dynamic feature when given
- do not infer Doppler, compressibility, posterior acoustic behavior, or artifact behavior unless source-supported or clearly labeled under the approved external mechanism clarification rule
- ultrasound cards should teach one sonographic retrieval target, not a generic modality paragraph

G5I) Nuclear medicine-specific integrity
- if the source supports nuclear medicine, PET/SPECT, scintigraphy, or tracer-specific normal appearance content, encode it using the existing card families and the nuclear medicine module
- nuclear medicine cards must name the tracer or exam when the source names it
- do not generalize one tracer's uptake, biodistribution, photopenia, washout, retention, or excretion behavior to another tracer
- do not convert source silence into a negative finding
- physiologic uptake, normal variants, and pseudolesions should be encoded when they prevent overcall or improve normal recognition
- nuclear medicine cards should teach one tracer-specific retrieval target, not a generic modality paragraph

H) Front-side blinding
- front fields must preserve the intended cognitive task
- no front may leak the answer through wording, captions, or extra fields
- image-containing non-UNKNOWN cards must keep Question blank

I) Field hygiene
- blank is better than fake content
- do not export filler
- do not export N/A
- do not export pseudo-diagnoses
- do not export fake differentials
- do not export pseudo-mechanisms
- do not export unsupported claims

If any gate fails:
- do not export TSV
- fix the violating note or split the note into separate card types
