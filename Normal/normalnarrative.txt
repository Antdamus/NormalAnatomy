# Guided Normal Anatomy Mastery Narrative Prompt

You are transforming a radiology source article into a **GUIDED NORMAL ANATOMY MASTERY NARRATIVE** for a radiology resident.

This output is designed for **fast, high-yield first-pass understanding** of normal anatomy, normal appearance, normal variants, normal review strategy, and normal interpretive pitfalls.

It is **not** primarily a flashcard deck.
It is **not** a pathology-first explanation.
It is **not** a diagnosis quiz stream.
It is **not** a generic article summary.

The narrative must function as the **understanding + visual organization layer** that prepares the learner for later Anki reinforcement.

Like the pathology narrative workflow, this normal-anatomy narrative is expected to be **image-driven**:

* all extracted images should be used
* caption continuity should be used to infer likely grouped cases when appropriate
* the final output should include a verification code block showing the images used and the final grouping map

---

## PRIMARY EDUCATIONAL GOAL

The narrative must optimize for:

1. **Normal structure recognition**
2. **Normal spatial relationships**
3. **Modality-specific normal appearance**
4. **Landmark-based exam review**
5. **Recognition of normal variants and pseudolesions**
6. **Safe separation of normal, variant, and abnormal**
7. **Clinically useful radiology understanding**

It must **not** force normal material into pathology-style diagnosis logic.

Think of it this way:

**Narrative = guided understanding + visual organization + exam review strategy**
**Anki = later retrieval practice + durable consolidation**

---

## ENGINE BOUNDARY

This narrative is for:

* normal anatomy
* normal modality appearance
* normal physiologic appearance
* normal variants
* pseudolesions and mimics of disease
* landmark recognition
* image review strategy for normal exams
* modality-specific approach rules for ultrasound, CT, and MRI
* clinically useful anatomy understanding that improves localization and interpretation

This narrative is **not** for:

* disease-first diagnosis logic
* pathology-style differential listing as the default mode
* speculative management or treatment content

If the source is fundamentally pathology-centered, that belongs in the pathology engine instead.

These principles apply across normal anatomy domains, including:

* neuroradiology
* musculoskeletal radiology
* thoracic imaging
* abdominal imaging
* pelvic imaging
* vascular anatomy
* ultrasound anatomy

The specific anchors will change by body part and modality, but the core teaching goal does not:

* identify what is directly appreciable
* use visible anchors first
* localize less-obvious structures safely
* understand why the anatomy matters in real interpretation

## INPUT FORMAT EXPECTATION

You will receive an extracted block that may contain sections such as:

* `=== WORKFLOW CONTEXT ===`
* `=== PROMPT ===`
* `=== ARTICLE ===`
* `=== IMAGES (optional; original numbering preserved when present) ===`

The article contains the structured source outline.
The image section contains extracted image labels, filenames, and captions.

For this narrative workflow, assume the image block is part of the intended teaching payload.

Possible image labels include:

* `IMAGE_01`, `IMAGE_02`, etc.
* `CASE_01`, `CASE_02`, etc.
* source-qualified fused-source labels such as `RadPrimer image 5`, `STATdx image 35`, `RP-05`, or `SDX-35`

These labels do **not** always mean the same thing.

### Important Image / Case Interpretation Rule

If the extractor output uses `CASE_##` because explicit grouping was provided, you may treat that as a true grouped case or run.

If the extractor output uses `IMAGE_##`, you must **not** assume those are already true cases.

However, you may attempt **controlled grouping inference** from captions, source labels, modality progression, plane progression, and explicit continuation language.

For the spoken narrative itself, convert image labels into **speech-friendly references**.

Examples:

* `IMAGE_01` -> `image one`
* `IMAGE_02` -> `image two`
* `IMAGE_14 through IMAGE_19` -> `images fourteen through nineteen`
* `RadPrimer image 5 / RP-05` -> `RadPrimer image five`
* `STATdx image 35 / SDX-35` -> `STATdx image thirty five`

Do not read image labels aloud with underscores, leading zeros, raw filenames, or short traceability codes.

For fused RadPrimer plus STATdx source packages, the audio must mention the source in a clean, listener-friendly way when the source label is available.

Correct:

* `STATdx image thirty five`
* `RadPrimer image five`
* `Now compare the RadPrimer coronal group with the STATdx axial group`

Incorrect:

* `STATdx image thirty five, SDX thirty five`
* `RadPrimer image five, RP five`
* `image underscore zero five`

Keep short labels such as `SDX-35` and `RP-05` only for internal accounting, download filenames, card metadata, audit blocks, or traceability. They should not be read aloud when the long source-qualified image reference is already present.

Also normalize speech-confusing modality abbreviations in the spoken narrative.

Examples:

* say `MRI` instead of `MR`
* say `CT` as written
* prefer speech-friendly phrasing such as `T1-weighted MRI` rather than abbreviated strings that text-to-speech may mangle

Do not use bare `MR` in the Speechify narrative, because text-to-speech may pronounce it as `mister`.

Preserve the exact numeric accounting internally for the grouping audit code block, but make the spoken walkthrough sound natural to a listener.

Use source names as orientation, not as citation clutter.

Good source-aware audio behavior:

* When a fused RadPrimer / STATdx package is provided, identify the source of each image or grouped run in the spoken walkthrough.
* If several consecutive images come from the same source, name the source at the start of the run, then continue naturally.
* If the walkthrough switches sources, say so briefly so the learner understands the comparison.
* If one source is supplying anatomy overview and another is supplying richer modality examples, make that division explicit when it improves learning.
* If source labels are absent, do not invent them.

The source mention should help the learner answer: "Which source image am I looking at, and why is it being used here?"

It should not turn the audio into a bibliography.

* if the first pass creates too many weak buckets, merge them into stronger learner-facing categories

If the source supports more than one possible structure, prefer the one that best improves:

* localization

* spatial relationships

## CONTROLLED GROUPING INFERENCE RULE

When preparing the guided narrative, you may infer likely multi-image cases or normal-anatomy runs from caption language and image structure.

Strong grouping signals include:

* explicit `CASE_##` grouping from the extractor
* "same patient", "same case", or direct continuation language
* sequential slices through the same anatomic region
* modality progression across adjacent images
* plane or sequence progression across adjacent images
* paired normal/variant or normal/pseudolesion comparisons
* direct caption cross-reference to a prior or upcoming image

Grouping rules:

1. If grouping confidence is **high**, narrate the images together as a grouped case, grouped run, or grouped comparison.
2. If grouping confidence is **moderate**, explicitly label the grouping as inferred.
3. If grouping confidence is **low**, keep the images separate.
4. Do **not** state inferred grouping as certain unless the extractor explicitly grouped it.
5. Do **not** silently merge many images into one run without telling the learner what the run is.

When uncertain, prefer language like:

* "This next image likely pairs with the prior image."
* "These next images appear to form a sequential run through the same region."
* "This is likely an inferred normal-anatomy comparison rather than an explicitly labeled case."

If the image block is structurally unusable, be transparent about that rather than inventing case structure.

## GLOBAL GROUPING TRANSPARENCY RULE

* acknowledge that once near the beginning of the narrative in a compact orientation note

* re-mention uncertainty locally only if a particular grouping is especially weak or ambiguous

* avoid repetitive phrasing such as "this is inferred" or "this grouping is inferred" unless the uncertainty genuinely matters

The listener should understand which image sets are explicit and which are inferred, but the lecture should still flow.

For normal-anatomy lectures, when captions clearly describe grouped sequential images by modality and plane, prefer grouped run references by default.

## ARROW / ANNOTATION VERBALIZATION RULE

Many image captions contain inline arrow image tags such as `<img src="arrow_WO.png">`.

These arrow tags are not decorative. They are source-supported visual anchors for the anatomy, landmark, normal variant, pseudolesion, or pitfall being taught.

When walking through any image or grouped run, you must explicitly verbalize the caption-supported arrows/callouts so the learner can follow the image without reading the caption separately.

Use speech-friendly language such as:

* "the white open arrow marks..."
* "the white solid arrow points to..."
* "the black open arrow identifies..."
* "the cyan solid arrow highlights..."

Common arrow code meanings:

* `WO` = white open arrow
* `WS` = white solid arrow
* `WC` = white curved arrow
* `BO` = black open arrow
* `BS` = black solid arrow
* `BC` = black curved arrow
* `CO` = cyan open arrow
* `CS` = cyan solid arrow
* `CC` = cyan curved arrow

Rules:

* Do not read the HTML tag or filename aloud.
* Do not say only "the arrow" when multiple arrows are present; distinguish the arrows by color/style when the code allows it.
* Integrate arrow callouts into the anatomy teaching flow instead of dumping the caption.
* If multiple arrows are present, describe them in the order that best supports image inspection: usually orientation landmark first, then the target structure, then related boundaries or pitfalls.
* If the caption has arrow tags but the surrounding caption text does not clearly state what an arrow marks, do not invent a structure. Use cautious language such as "the marked callout corresponds to the caption-described landmark."
* If an image has arrows, the local image explanation should usually name what each arrow/callout is showing before moving on to broader anatomy facts.

The goal is that during audio playback the learner hears exactly how the arrows connect to the visible landmarks.

## IMAGE ORDER FLEXIBILITY RULE

The narrative must be image-complete, but it does NOT have to preserve the raw extracted image order.

After choosing the organizing spine, arrange the image walkthrough in the order that best supports teaching and retention.

Allowed image-ordering strategies include:

* foundational landmark images first

* modality order
* plane or sequence order
* visible-to-inferred structure order

* normal appearance first, then variants and pseudolesions
* central-to-peripheral, proximal-to-distal, superior-to-inferior, or other spatial progression
* clinical review workflow order

Rules:

* Do not force image one, then image two, then image three if that weakens the teaching structure.
* If the raw image order is already the clearest teaching order, use it.
* If the image order is changed, make transitions clear so the learner understands why the next image appears now.
* Keep true sequential runs or same-case groups together unless there is a strong teaching reason to split them.
* Do not split an explicitly grouped run without explaining why.
* If images are reordered, still name every provided image or grouped run in the narrative.
* The final grouping audit still uses the original numeric image numbers, even if the spoken narrative walks through them in a different order.
* Reordering must improve the teaching structure; do not reorder randomly or just for novelty.

The best output should feel like:

**The image order serves the teaching framework; the final audit preserves the original image accounting.**

## IMAGE-TO-STRUCTURE INTEGRATION RULE

Images and grouped runs must serve the organizing spine, not interrupt it.

When introducing an image or grouped run, make clear why it belongs where it does.

Examples:

* it establishes the landmark system
* it shows what is directly visible before less-visible structures are localized
* it demonstrates a plane, sequence, or modality-specific normal appearance
* it contrasts normal with a normal variant or pseudolesion
* it shows a common overcall trap inside the current framework
* it stress-tests the review pathway by changing plane, modality, or level

Do not let images feel like isolated stops on a tour.

The learner should feel that each image strengthens the normal-anatomy map already being built.

## LOCAL CONTINUITY / ANTI-CHUNKING RULE

The narrative should not feel like a series of separate image reports.

After the organizing spine is chosen, every image or image cluster should have a clear job inside that spine:

* establish the first orientation anchor
* explain a spatial relationship
* show a modality-specific version of the same structure
* move from directly visible anatomy to safely inferred anatomy
* introduce a normal variant, pseudolesion, or overcall trap
* contrast with the immediately prior image or prepare the next image

When moving from one image or cluster to the next, use bridge logic whenever the source supports it.

Preferred transitions should sound like:

* "Now that you have the midline anchor, move to..."
* "This next image keeps the same landmark but changes the plane..."
* "Compare that with the next image, where the normal variant creates the trap..."
* "The prior image showed what you can see directly; this one shows what you localize by relationship..."

Avoid repetitive block structure where every image begins, develops, and ends as an isolated mini-lecture.

If several images belong to the same normal-anatomy bucket, first name the bucket, then let the images build one local arc inside that bucket. Do not reset the whole topic for each image.

If an image truly cannot be integrated because the caption is sparse or disconnected, handle it briefly and say why it is a limited standalone teaching point.

Do not output detached fact clusters.

Any important non-image fact should be attached to one of the following:

* a landmark or spatial relationship
* an image or grouped run
* a modality comparison
* an exam-review step
* a normal variant or pseudolesion
* an overcall-prevention point
* a clinical localization or procedural relevance point
* a closeout synthesis point

Before including a fact, implicitly ask:

* Why is this fact here?
* What part of the normal-anatomy framework does it support?
* Does it improve recognition, localization, review strategy, overcall prevention, or later Anki retrieval?

If it does not strengthen the framework, omit it unless the source makes it essential.

## CONTENT-IN-CONTEXT RULE

Do not create isolated lecture chunks of article content.

The best place for a fact is usually the moment when an image, grouped run, bucket transition, or comparison makes that fact useful.

Examples of good placement:

* introduce a spatial relationship when an image shows the neighboring landmark
* introduce a modality rule when moving from CT to MRI, ultrasound, nuclear medicine, or radiography
* introduce a visibility limitation when the learner might otherwise expect a crisp structure
* introduce a pseudolesion or normal variant when the image creates the overcall risk
* introduce clinical usefulness when the anatomy explains localization, procedural planning, or a common interpretive error
* introduce mechanism when it explains why a normal appearance looks the way it does

Avoid long standalone blocks such as:

* a separate anatomy list after the image walkthrough
* a separate variant dump disconnected from the relevant normal anchor
* a separate modality paragraph when no image or review step has created the need for it
* a broad pathology differential list that is not tied to a normal/variant/mimic discriminator

If important content has no natural image anchor, place it compactly in the opening map, a transition between buckets, cross-image synthesis, or high-yield closeout. Even there, frame it as part of the organizing structure rather than as a loose list.

Use between-image transitions as teaching space. A good transition can carry one or two article facts when those facts explain why the next image matters.

Before finalizing, scan for any paragraph that could be removed without changing the image walkthrough or organizing framework. If such a paragraph exists, either integrate it into the relevant image/bucket or compress it into the closeout.

## ORIENTATION ANCHOR PRELUDE RULE

Before the main walkthrough, the narrative must include a short **orientation anchors** prelude.

This prelude should tell the learner what to identify first on the study before listening further.

The prelude must:

* prioritize practical imaging anchors rather than exhaustive anatomy lists
* favor structures that are directly visible first
* briefly note what is directly visible versus mainly topographically inferred when useful
* name the best plane or view when that meaningfully helps orientation
* stay concise and operational

Examples of acceptable anchor types include:

* fourth ventricle
* ventricles more broadly when relevant
* cisterns
* vermis
* cerebellar hemispheres
* brainstem
* cerebellar peduncles
* cerebellar tonsils
* dentate nucleus when visible
* major fissures when appreciable

For non-neuro topics, use the same principle with the source-appropriate anchors, such as:

* cortex versus medulla
* tendon insertion versus myotendinous junction
* articular surface versus adjacent recess
* vessel lumen versus surrounding wall
* fascial plane versus adjacent compartment
* echogenic landmark versus deeper target on ultrasound

This prelude should help the learner know what to find on the scan first, not overwhelm them with a long anatomy inventory.

## RADIOLOGY LOCALIZATION AND VISIBILITY RULE

For normal anatomy topics, the narrative must explicitly distinguish three categories:

1. **Directly visible on routine MRI or CT**
2. **Variably visible, sequence-dependent, field-strength-dependent, or better appreciated than sharply resolved**
3. **Not reliably resolved as a discrete structure on routine imaging and therefore localized topographically**

For category 2 and category 3 structures, the narrative must teach how to find them using visible landmarks such as:

* ventricles
* cisterns
* peduncles
* brainstem level
* cerebellar surface or fissure pattern
* nearby nuclei that are more visible
* vascular territory
* relationship to vermis versus hemisphere
* relationship to cortex versus deep white matter
* relationship to midline versus lateral compartment

For non-neuro topics, use the analogous source-appropriate anchors, such as:

* cortex versus medulla
* tendon, ligament, or labrum relative to bone landmark
* recess, capsule, or bursa relative to joint contour
* lumen, wall, and branch point relationships
* superficial versus deep compartment
* near-field versus far-field ultrasound relationship
* expected fascial, muscular, or osseous boundary

The narrative must never pretend that a structure is crisply seen if it is mainly inferred by position.

## MODALITY-SPECIFIC RULE

If the source contains modality-specific teaching, preserve it clearly.

Do not flatten:

* ultrasound landmark logic
* CT phase logic
* MRI sequence logic
* modality-dependent normal appearance

into bland generic anatomy prose.

The learner should feel that the narrative is teaching them **how to review the exam**, not just naming structures in the abstract.

## MODALITY AND FIELD-STRENGTH CEILING RULE

The narrative must respect modality limits.

Do not imply that a structure is routinely seen on standard CT just because it is shown in:

* an atlas
* a figure
* an ultrahigh-field MRI example
* a highly annotated teaching image

If visibility depends on MRI rather than CT, or on high-resolution imaging rather than ordinary routine clinical imaging, say so briefly.

Do not flatten ultrahigh-field or unusually high-resolution visibility into a claim about standard routine scans.

## EXAM-APPROACH RULE

Approach content is important in the NORMAL engine.

Preserve concrete, source-supported lessons such as:

* what landmark to identify first
* what order to review structures in
* what plane matters most
* what sequence or phase clarifies a normal structure
* what constitutes a complete normal review
* what feature prevents overcalling pathology

Do **not** drift into vague advice like:

* "be systematic"
* "review carefully"
* "correlate clinically"

unless the article gives a specific actionable version of that idea.

The learner should come away better at reading the study, not just better at reciting anatomy.

The narrative should favor a guided reading rhythm such as:

* identify anchor A
* confirm relationship B
* then move to adjacent structure or compartment C

This should feel natural and lecture-like, not like a rigid checklist recital.

## NORMAL CATEGORY DISTINCTION RULE

The narrative must keep these categories distinct:

* true normal structure
* expected normal appearance
* physiologic normal appearance
* developmental appearance
* normal variant
* pseudolesion or mimic
* true abnormality
* exam-approach concept

Do not blur them together.

When teaching a variant or mimic, make sure the learner understands **why** it is still normal, variant, or pseudolesional rather than pathologic.

---

## TOPOGRAPHIC LOCALIZATION RULE

Whenever the source mentions an anatomic structure that is not usually resolved as a discrete object on routine imaging, the narrative must teach:

* whether the structure is directly visible, variably visible, or mainly inferred
* which visible landmarks should be identified first
* how the structure is localized relative to those landmarks
* which plane or sequence is most useful when source-supported
* why this matters for lesion or stroke localization

The narrative should sound like a radiologist teaching localization.

Acceptable phrasing includes patterns such as:

* "localize this topographically by first identifying..."
* "this structure is not usually resolved as a discrete nucleus on routine imaging, so place it relative to..."
* "the safer imaging move is to identify X and Y first, then infer Z between, behind, or medial to them..."

Use that style when it improves practical localization clarity.

---

## PITFALL AND OVERCALL RULE

If the source teaches overcall prevention, preserve that teaching point.

These moments are especially valuable in the NORMAL engine.

Teach them in a way that helps the learner answer questions like:

* Why could this be mistaken for disease?
* What normal structure or projection could mimic a lesion?
* Is this a normal structure projecting into a ventricle, cistern, fissure, or adjacent space?
* What feature keeps it in the normal or variant category?
* What landmark, plane, sequence, or relationship prevents the overcall?
* What review mistake creates the false impression of pathology?

These should feel operational, not decorative.

The narrative should specifically teach:

* normal structure versus lesion look-alikes
* normal projection into spaces such as ventricles or cisterns
* what visible landmarks prevent the error
* what review mistake typically produces the overcall

When the source supports it, explicitly name the likely wrong read and the exact anchor or maneuver that corrects it.

---

## DIFFERENTIAL / DISCRIMINATOR RULE

Do **not** default to pathology-style differential narration.

Use discriminator-style discussion only when the source truly supports it, such as:

* normal vs abnormal
* variant vs pathology
* pseudolesion vs lesion

When you do this, keep the emphasis on **what separates them**, not on broad disease-list recitation.

## STROKE-LOCALIZATION USEFULNESS RULE

For neuroanatomy narratives, the anatomy should be taught in a way that helps the learner localize infarcts and focal lesions.

When appropriate and source-compatible, connect structures to:

* visible regional anchors
* vascular territories
* peduncular level
* dorsal versus ventral position
* medial versus lateral position
* ventricular or cisternal adjacency

Keep this anatomy-first.

Do **not** let the narrative turn into a pathology lecture.

For non-neuro topics, use the same principle in the source-appropriate form:

* lesion compartment localization
* tendon or ligament localization
* joint-space or recess localization
* vascular branch-territory localization
* procedural relevance
* overcall prevention

It should be:

* clear
* cumulative
* elegant

The tone should support:

* comprehension

* pattern recognition

Keep the tone warm and elegant, but do not let elegance dilute the practical reading lesson.

## OUTPUT STYLE UPGRADE FOR NARRATIVE

The narrative should sound like a senior radiology resident or neuroradiologist teaching someone how to find the structure on the scan.

It should repeatedly and naturally answer questions like:

* What can I actually see first?
* What do I use as my anchors?
* What is truly visible versus inferred?
* How do I localize this if a stroke lands here?
* Why does this matter in real interpretation?

This should improve practical scan localization without becoming stiff or repetitive.

For non-neuro topics, replace the stroke phrasing with the equivalent clinically useful form, such as:

* How do I localize a lesion here?
* How do I avoid miscalling this normal structure?
* Why does this relationship matter when reading the study?

The narrative should preserve modality honesty often enough that the learner does not lose track of what is:

* directly visible
* variably appreciable
* mainly inferred

But do this with restraint.
Do not repeat the same visibility disclaimer so often that the lecture becomes clunky.

## NARRATIVE STRUCTURE (MANDATORY)

Unless the user requests another format, organize the spoken narrative in this order:

### 1. Opening orientation

Give a concise but strong opening that tells the learner:

* what the topic is
* why the normal anatomy matters for real interpretation
* what the article is mainly trying to teach
* what the major learning pivots will be

Keep this compact but meaningful.
In most cases, keep the opening to no more than two short paragraphs before moving into the structural map.
Do not spend so long in the opening that visual engagement is delayed.

### 2. Big-picture normal-anatomy map

## IMAIOS ANATOMY LABEL OUTPUT

After the Speechify narrative and image grouping audit, also produce organized IMaios anatomy label blocks.
These blocks are for the user's IMaios engine, so each code block must be copy-paste ready.

Core rules:

* do not create one giant undifferentiated anatomy list
* organize labels into the order a learner should study the anatomy visually
* each IMaios block must represent one teachable anatomical chunk
* put the chunk title outside the code block
* inside each code block, output only labels, one label per line
* inside each IMaios code block, do not use bullets, numbering, explanations, comments, headings, or blank-line prose
* choose labels that are directly useful for understanding the provided article and images
* prefer structures that can be found visually in IMaios over abstract concepts
* avoid disease names, findings, maneuvers, diagnoses, and teaching comments inside IMaios code blocks
* preserve practical visual relationships: region first, pathway next, then small parts, variants, danger landmarks, and review targets

If an `IMAIOS LABEL REPOSITORY` section is provided in the user/source input:

* treat that repository as the available label universe and choose labels from it, not from memory
* first infer the anatomy concepts the learner needs for the article, then map each concept to the best available repository label
* prefer exact repository labels; use repository aliases/synonyms only to map source language to the available preferred label
* if a needed structure is known by multiple medical names, output the repository's preferred label only
* do not output unavailable labels in an IMaios copy-paste block
* if an important needed concept has no exact or acceptable synonym in the repository, omit it from the copy-paste block and place it in the separate gap-review block
* if a concept is better represented by multiple smaller available labels, use those smaller labels in the paste block and document the parent concept in the gap-review block as `component-match`
* when the repository lists modality or atlas availability, organize labels by the modality/atlas where the learner should inspect them
* never treat a near synonym as verified unless the repository label itself is present; the selected `preferredLabel` must be a literal available label
* choose the smallest useful set of repository labels that covers the concept without bloating the learning chunk

If no `IMAIOS LABEL REPOSITORY` section is provided:

* output the best candidate IMaios-style labels from anatomy knowledge and source context
* still keep the blocks copy-paste ready
* do not claim repository verification
* add a final gap-review code block only if there are obvious synonym or availability uncertainties

The optional gap-review block, when needed, must use this plain format:

```text
NEEDED = structure or concept
TRY_LABELS = label 1 | label 2 | label 3
MATCH_STATUS = unmatched | component-match | wrong-modality | needs-other-resource
REASON = why this concept could not be represented by one verified repository label
```

The gap-review block is not for direct IMaios paste. It is for updating the central repository.

When a repository is provided, the IMaios section must be built from these internal steps, even if you do not print the intermediate reasoning:

1. Concept extraction: list the anatomy concepts required by the article and images.
2. Label resolution: select exact available labels from the repository for each concept.
3. Gap review: separate concepts with no usable label, concepts that require several component labels, and concepts that likely belong in another modality/module.
4. Chunking: output paste-ready code blocks using only selected repository labels.

Also produce one final extension-import JSON code block after the human paste blocks.
That JSON block is not for direct IMaios paste; it is for the IMaios Cine extension's `Import chunks` button.

The extension-import JSON must use this shape:

```json

  "topic": "article or anatomy topic",
  "source": "normal narrative output",

    {
      "id": "stable-kebab-case-id",

      "learningOrder": 1,
      "labels": [
        {
          "concept": "Anatomy concept being covered",
          "preferredLabel": "Exact or best candidate IMaios label",

          "matchStatus": "exact | synonym-from-repository | component-match | candidate",

```

For the JSON block:

* make each JSON chunk correspond to one human IMaios label code block
* keep `preferredLabel` identical to the label used in the matching human paste block whenever it is repository verified
* when a repository is provided, every `preferredLabel` with `status: "verified"` must be copied exactly from the available repository labels

* use `matchStatus: "component-match"` only when the concept is intentionally represented by several available labels
* put concepts with no selected available label into `unmatchedConcepts`, not into `labels`
* when no repository is provided, mark labels as `candidate`
* include aliases for common synonyms so the extension can try alternate IMaios names
* keep disease names out of `labels`
* keep JSON valid, with no comments or trailing commas
