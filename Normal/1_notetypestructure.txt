-----------------------------------------------------

SHARED CHASSIS RULE

The NORMAL engine must reuse the same:
- note type
- field order
- TSV structure
- media pipeline
- import workflow
- ID logic
- tag architecture

Do NOT invent new fields.
Do NOT reorder fields.
Do NOT create a second note type.

The NORMAL engine changes field usage and content rules, not the chassis.

Localization support must therefore be mapped into existing explanatory fields such as:
- Imaging_Differentiation
- Mechanism
- Boards_Trap
- High_Yield_A

Do not create new fields just to hold visibility or localization teaching.

-----------------------------------------------------

FIELD-USAGE PRINCIPLE

Each field may be:
- required
- conditionally allowed
- forbidden for a given card instance

Do NOT force content into a field just because the field exists.

If a field does not naturally fit the learning goal, leave it blank.
Blank is preferred over fake content.

-----------------------------------------------------

GLOBAL FIELD INTERPRETATION FOR THE NORMAL ENGINE

Question
- image-based recognition prompt for a structure, relationship, appearance, variant, mimic, or landmark

Entity_Label
- the true target entity for the card

Imaging_Differentiation
- why the appearance is normal, why it is a variant, what keeps it from being overcalled, or what discriminator matters most
- visibility classification when useful
- practical landmark-based localization support
- concise "seen directly vs inferred topographically" teaching

Mechanism
- why a normal appearance looks the way it does, why a sequence or phase matters, or why a variant exists
- why a localization rule works when the source explicitly supports the reason

Boards_Trap
- a normal mimic, pitfall, pseudolesion, or overcall-protection rule
- including what visible landmarks prevent a mistaken lesion call

High_Yield_Q / High_Yield_A
- compact definition, landmark rule, approach rule, checklist, modality anchor, measurement rule, or variant anchor
- orientation-anchor and localization-rule content when image recognition is not the main task

Radiopaedia fields
- reserved for the future Radiopaedia-specific workflow only

-----------------------------------------------------

SINGLE FRONT FAMILY RULE

Only ONE front-driving prompt family may exist on a note.

Front-driving prompt families are:
- Question
- Differential_Q
- Mechanism_Q
- Boards_Trap_Q
- High_Yield_Q

If more than one of these is populated on the same note:
- the note is structurally invalid
- split the content into separate notes

-----------------------------------------------------

GLOBAL COMPANION-FIELD DISCIPLINE

Companion fields must not appear alone.

Therefore:
- Differential_Q must be blank unless Differentials is populated
- Mechanism_Q must be blank unless Mechanism is populated
- Boards_Trap_Q must be blank unless Boards_Trap is populated
- High_Yield_A must be blank unless High_Yield_Q is populated
- Image_Annotated must be blank unless Image is populated
- Original_Caption must be blank unless Image is populated

If any orphan companion field appears:
- the note is structurally invalid
- clear the orphan field or complete the intended card type

-----------------------------------------------------

FIELD RULES

1) Clinical_Context
Required always.

Allowed forms:
- short neutral case context for an image-based card
- age/sex only when source-supported and useful
- modality or exam context only when needed for the learning task
- a unique short standalone code for a concept-only card when no real clinical context exists

Do NOT fabricate history.

2) Image
Required for image-based cards.
Optional for concept-only cards.

Use for:
- structure recognition
- relationship recognition
- modality appearance
- normal variants
- pseudolesions
- landmark identification

3) Image_Annotated
Conditionally allowed only when Image is populated.

If populated:
- preserve image order exactly
- remain source-faithful
- do not add unsupported interpretation

Annotated-only image notes are not supported by this engine.

4) Question
Reserved for NORMAL UNKNOWN.

If Question is populated:
- Image must also be populated
- no other front-driving prompt family may be populated

5) Most_Likely_Diagnosis
Forbidden by default in the NORMAL engine.

Allowed only when a tightly bounded mimic-rejection design truly requires a diagnosis-style anchor.
Do NOT insert fake answers such as "normal study" or "none."

6) Entity_Label
High-value field in this engine.
Use a concise target entity, not an essay.

7) Differential_Q
Conditionally allowed only when Differentials is populated.

8) Differentials
Populate only when intentionally creating NORMAL DIFFERENTIAL DRILL.

9) Imaging_Differentiation
Conditionally allowed when the card type explicitly permits it.

Use for:
- why this is normal
- why this is a variant
- what separates the mimic from pathology
- what feature or step makes the review safe

10) Original_Caption
Conditionally allowed only when Image is populated and the image workflow requires it.
Do not paraphrase a verbatim caption field.

11) Mechanism_Q
Conditionally allowed only when Mechanism is populated.

12) Mechanism
Populate only when intentionally creating NORMAL MECHANISM.

13) Boards_Trap_Q
Conditionally allowed only when Boards_Trap is populated.

14) Boards_Trap
Populate only when intentionally creating NORMAL BOARDS TRAP.

15) High_Yield_Q
Populate only when intentionally creating NORMAL HIGH-YIELD.

16) High_Yield_A
Conditionally allowed only when High_Yield_Q is populated.
Keep compact, structured, and source-faithful.

17-20) Radiopaedia fields
Leave blank unless intentionally generating the future RADIOPAEDIA DRILL workflow.

21) summary
Required only for cards derived from a full RadPrimer or STATdx article.
Blank otherwise.

-----------------------------------------------------

FIELD HYGIENE RULE

Never populate a field with:
- filler
- repetition
- fake differentials
- fake diagnoses
- pseudo-mechanisms
- unsupported claims
- "N/A"

Blank is better than false structure.
