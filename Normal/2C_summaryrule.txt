SECTION PURPOSE

This file defines when the summary field is required and exactly how it must be written for the NORMAL engine.

-----------------------------------------------------

WHEN SUMMARY IS REQUIRED

The summary field MUST be populated for all cards derived from:
- a full RadPrimer article
- a full STATdx article

This applies regardless of card type:
- NORMAL UNKNOWN
- NORMAL DIFFERENTIAL DRILL
- NORMAL MECHANISM
- NORMAL BOARDS TRAP
- NORMAL HIGH-YIELD

If the card is not derived from a full RadPrimer or STATdx article:
- leave summary blank

-----------------------------------------------------

SUMMARY SOURCE DISCIPLINE

The summary must be derived ONLY from the full anchor article.

The summary must:
- reflect full-article review
- be article-level rather than card-level
- remain identical across all cards generated from the same article
- remain source-faithful
- stay inside the article's supported scope
- avoid outside knowledge
- avoid Core supplementation
- avoid Radiopaedia supplementation
- avoid unsupported inference

The summary is an auxiliary structured synthesis.
It is NOT the source of truth for card generation.
Cards must still come from the full article itself.

-----------------------------------------------------

SUMMARY FORMAT (STRICT)

The summary field must contain:
- no literal newline characters
- one continuous Anki-safe HTML string
- no raw markdown markers such as #, ##, **, *, -, or markdown tables

Use simple Anki-safe HTML only:
- <b>
- <br>
- <div>
- <ul>
- <ol>
- <li>
- light inline style attributes only when they improve hierarchy and remain robust in Anki

The summary must begin with:
- a visually distinct title block using the source article's actual title or main topic
- the title block should include a clean emoji or icon anchor
- then 1-2 sentences summarizing the main topic from the article only

SUMMARY ICON ENCODING RULE

For TSV export, do not write literal emoji characters directly into any field, including the summary field.

When the summary requires emoji or icon anchors, encode them as HTML numeric character references instead of literal Unicode characters.

Examples:
- Use `&#x1F9ED;` instead of the compass emoji
- Use `&#x1F50D;` instead of the magnifying glass emoji
- Use `&#x26A0;&#xFE0F;` instead of the warning emoji
- Use `&#x2B50;` instead of the star emoji

This prevents PowerShell, console, clipboard, or intermediate script encoding from converting icons into question marks during TSV generation.

The summary should still render with icons in Anki/HTML, but the TSV text itself should remain ASCII-safe.

The title block should:
- stand out visually from the rest of the summary
- use simple HTML hierarchy such as a bold title inside a <div>
- avoid hardcoded example entities
- allow light icon anchors only when they genuinely improve scanability, and encode them as HTML numeric character references

Then use clearly labeled sections only when supported by the article.

The summary should read like a polished radiology study handout:
- visually readable first
- easy to scan
- not plain or paragraph-heavy
- organized with clear spacing between major sections

Formatting hierarchy rules:
- major section headers must be on their own line
- subsection headers must also be on their own line
- explanatory content should appear underneath as bullets or nested bullets
- do NOT use inline `Header: long sentence` formatting for the main outline
- after the initial 1-2 sentence summary, prefer bullet-based and hierarchical formatting over prose blocks
- when choosing between fewer longer bullets versus more shorter nested bullets, prefer more shorter nested bullets

Preferred section labels:
- <b>Abbreviations / Acronyms</b> when source-supported
- <b>Summary</b>
- <b>Key Imaging Findings</b>
- <b>Differential Diagnosis</b>
- <b>Clinical Presentation</b>
- <b>Pearls / Pitfalls</b>
- <b>Key Measurements / Criteria / Scoring Systems</b>
- other article-supported labels when needed for readability in the NORMAL workflow, such as landmarks, structures, review approach, variants, discriminators, or modality anchors

Preferred section order when supported by the article:
- Abbreviations / Acronyms
- overview
- Key Imaging Findings
- Differential Diagnosis
- Clinical Presentation / Demographics
- Pearls / Pitfalls / Diagnostic Checklist
- Super Summary

Major sections should use tasteful emoji or icon anchors for readability.
Use varied, clean icons with restraint.
Every major section header should use a visible emoji or icon anchor, encoded for TSV export as an HTML numeric character reference rather than as a literal emoji character.
Do not use the same icon for every section unless necessary.
Permission is granted to vary icons or color-style icons for readability and scanability.

Allowed emoji / icon palette for title and major section headers:

The palette defines the intended rendered icons. During TSV generation, these icons must be emitted as HTML numeric character references, not literal Unicode emoji.

Examples from the allowed palette:
- colored squares/circles: `&#x1F7E5;`, `&#x1F7E6;`, `&#x1F7E8;`, `&#x1F7E9;`, `&#x1F7EA;`, `&#x1F7E7;`, `&#x2B1B;`, `&#x2B1C;`, `&#x1F534;`, `&#x1F535;`, `&#x1F7E2;`, `&#x1F7E1;`, `&#x1F7E3;`, `&#x1F7E0;`, `&#x26AB;`, `&#x26AA;`
- common radiology-study icons: `&#x1F9E0;`, `&#x1F511;`, `&#x2696;&#xFE0F;`, `&#x1F4A1;`, `&#x1F9FE;`, `&#x1F4CC;`, `&#x1F4DA;`, `&#x1F4DD;`, `&#x1F50D;`, `&#x1F4D6;`, `&#x1F9EC;`, `&#x1FA7B;`, `&#x1F5BC;&#xFE0F;`, `&#x1F4CA;`, `&#x1F9ED;`, `&#x1F6A9;`, `&#x2705;`, `&#x2B50;`, `&#x26A0;&#xFE0F;`, `&#x1F9EA;`, `&#x1F3F7;&#xFE0F;`, `&#x1F4CD;`, `&#x1F3AF;`, `&#x1F6E0;&#xFE0F;`, `&#x1F52C;`, `&#x1F5C2;&#xFE0F;`, `&#x1F4CE;`, `&#x2733;&#xFE0F;`, `&#x1F536;`, `&#x1F537;`, `&#x2666;&#xFE0F;`, `&#x25AA;&#xFE0F;`

Within Key Imaging Findings:
- group findings by modality when the article supports modality-specific organization
- do not flatten modality-specific findings into one mixed list
- modality subsection titles must be on their own lines with bullets beneath them

Preferred imaging subsections when supported:
- <b>Radiography / Best Initial Test</b>
- <b>CT</b>
- <b>MR Imaging</b>
- <b>Ultrasound / Other</b>
- <b>Distribution / Hallmark Signs</b>

End with:
- a concise <b>Super Summary</b> section

Use bullets generously for readability.
Prefer compact, high-yield bullets rather than long prose bullets.
Each bullet should ideally communicate one radiology takeaway.
Use nested bullets when helpful to split longer concepts into smaller reviewable chunks.
Omit unsupported sections rather than fabricating them.

-----------------------------------------------------

SUMMARY CONTENT RULES

The summary should capture only article-supported material that would help explain why the resulting cards exist.

It should prioritize radiology study usefulness, especially:
- imaging findings
- high-yield radiology anchors
- measurements
- criteria
- scoring systems
- discriminators
- pitfalls

Elevate into dedicated bullets or subsections when source-supported:
- article-defined abbreviations or acronyms
- hallmark signs
- favored locations or distributions
- measurements
- criteria
- scoring systems
- diagnostic pivots
- review checklists

It should NOT:
- become a pathology-style differential paragraph
- become a freeform essay
- mix article content with textbook fill-in
- include claims absent from the anchor article

-----------------------------------------------------

SUMMARY COMPRESSION RULE

Summaries should be dense and operational.

Use:
- concise bullets
- article-supported separators
- article-supported approach steps
- article-supported pitfalls
- strong visual hierarchy
- spacing between major sections
- visually distinct headers and, when useful, subsection headers
- hierarchical bullets with nested sub-bullets when needed for readability

Do not waste summary space on generic statements.
