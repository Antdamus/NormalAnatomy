Allowed Card Types:

A) UNKNOWN
B) DIFFERENTIAL DRILL
C) MECHANISM
D) BOARDS TRAP
E) HIGH-YIELD
F) RADIOPAEDIA DRILL (to be implemented later)

No other card types allowed.

-----------------------------------------------------

Card-Type Trigger Rules (Field -> Card Type Mapping):

- UNKNOWN is generated ONLY when BOTH:
    - Image field is populated
    AND
    - Question field is populated.

- DIFFERENTIAL DRILL is generated when the Differentials field is populated.

- MECHANISM is generated when the Mechanism field is populated.

- BOARDS TRAP is generated when the Boards_Trap field is populated.

- HIGH-YIELD is generated when the High_Yield_Q field is populated.

- RADIOPAEDIA DRILL will be generated later when Radiopaedia-specific fields are intentionally populated.

-----------------------------------------------------

Trigger Safety Enforcement:

- If Image is populated but Question is EMPTY -> the card must NOT be treated as UNKNOWN.
  (This allows images to safely exist in Boards Trap or Differential Drill.)

- Any non-UNKNOWN card that includes Image MUST keep Question empty.

- Fields must NOT be populated unless that card type is intentionally being created.

- No unintended card types may be generated.

-----------------------------------------------------

Summary Field Rule (GLOBAL - CRITICAL):

The summary field MUST be populated for ALL cards that originate from:
    - RadPrimer
    - STATdx

This applies regardless of card type:
    - UNKNOWN
    - DIFFERENTIAL DRILL
    - MECHANISM
    - BOARDS TRAP
    - HIGH-YIELD

-----------------------------------------------------
SUMMARY SOURCE DISCIPLINE (MANDATORY)

The summary field MUST be derived ONLY from the FULL specific RadPrimer/STATdx article or case provided.

The summary MUST:
- Reflect full-article review, not a short digest, excerpt, or previously compressed summary
- Be based on the entire provided article/case text relevant to the topic
- Be article-level rather than card-level
- Remain identical across all cards generated from the same article/case bundle
- NOT mix in Core Radiology content
- NOT add outside knowledge
- NOT include inferred material
- NOT be generated from the Core validation digest
- NOT be generated from the Core Summary
- NOT substitute for full-source review

The summary is an auxiliary structured synthesis.
It is NOT the source of truth for card generation.
Cards must be generated from the full article itself.

-----------------------------------------------------
SUMMARY PURPOSE (QUALITY STANDARD)

The summary is a high-yield reconstruction aid, not a terse note fragment.

Its purpose is to:
- Preserve the major teaching architecture of the article
- Capture the article's most exam-relevant distinctions and discriminators
- Retain high-value pearls, pitfalls, and separators
- Provide enough detail that the main teaching points of the article can be recovered without reopening it

Over-compression that removes meaningful distinctions is prohibited.

-----------------------------------------------------
SUMMARY FORMAT (STRICT)

CRITICAL:
- Output must contain NO literal newline characters
- Output must be ONE continuous Anki-safe HTML string
- Do NOT emit raw markdown markers such as #, ##, **, *, -, or markdown tables

Use simple Anki-safe HTML only:
- <b>
- <br>
- <div>
- <ul>
- <ol>
- <li>
- light inline style attributes only when they improve hierarchy and remain robust in Anki

The summary must begin with:
- a visually distinct title block using the source article's actual title or main topic
- the title block should include a clean encoded icon anchor
- then 1-2 sentences summarizing the main topic from the source article/case only

SUMMARY ICON ENCODING RULE

For TSV export, do not write literal emoji characters directly into any field, including the summary field.

When the summary requires emoji or icon anchors, encode them as HTML numeric character references instead of literal Unicode characters.

Examples:
- Use `&#x1F9ED;` instead of the compass emoji
- Use `&#x1F50D;` instead of the magnifying glass emoji
- Use `&#x26A0;&#xFE0F;` instead of the warning emoji
- Use `&#x2B50;` instead of the star emoji

This prevents PowerShell, console, clipboard, browser-copy, or intermediate script encoding from converting icons into question marks during TSV generation.

The summary should still render with icons in Anki/HTML, but the TSV text itself must remain ASCII-safe.

The title block should:
- stand out visually from the rest of the summary
- use simple HTML hierarchy such as a bold title inside a <div>
- avoid hardcoded example entities
- allow light icon anchors only when they genuinely improve scanability, and encode them as HTML numeric character references

Then use clearly labeled sections only when supported by the source article/case.

The summary should read like a polished radiology study handout:
- visually readable first
- easy to scan
- not plain or paragraph-heavy
- organized with clear spacing between major sections

Formatting hierarchy rules:
- major section headers must be on their own line
- subsection headers must also be on their own line
- explanatory content should appear underneath as bullets or nested bullets
- do NOT use inline `Header: long sentence` formatting for the main outline
- after the initial 1-2 sentence summary, prefer bullet-based and hierarchical formatting over prose blocks
- when choosing between fewer longer bullets versus more shorter nested bullets, prefer more shorter nested bullets

Preferred section labels:
- <b>Abbreviations / Acronyms</b> when source-supported
- <b>Summary</b>
- <b>Key Imaging Findings</b>
- <b>Differential Diagnosis</b>
- <b>Clinical Presentation</b>
- <b>Pearls / Pitfalls</b>
- <b>Key Measurements / Criteria / Scoring Systems</b>
- <b>Clinical / Path / Mechanism</b> when source-supported
- <b>Board-Relevant Pivots</b> when source-supported

Preferred section order when supported by the article:
- Abbreviations / Acronyms
- overview
- Key Imaging Findings
- Differential Diagnosis
- Clinical Presentation / Demographics
- Pearls / Pitfalls / Diagnostic Checklist
- Super Summary

Major sections should use tasteful encoded icon anchors for readability.
Use varied, clean icons with restraint.
Every major section header should use a visible encoded icon anchor, emitted for TSV export as an HTML numeric character reference rather than as a literal emoji character.
Do not use the same icon for every section unless necessary.
Permission is granted to vary icons or color-style icons for readability and scanability.

Allowed encoded icon palette for title and major section headers:

The palette defines the intended rendered icons. During TSV generation, icons must be emitted as HTML numeric character references, not literal Unicode emoji.

Do not copy, emit, or rely on any literal/non-ASCII legacy icon samples that may appear in older prompt copies. Only entity-encoded icon anchors are valid for TSV output.

Examples from the allowed palette:
- colored squares/circles: `&#x1F7E5;`, `&#x1F7E6;`, `&#x1F7E8;`, `&#x1F7E9;`, `&#x1F7EA;`, `&#x1F7E7;`, `&#x2B1B;`, `&#x2B1C;`, `&#x1F534;`, `&#x1F535;`, `&#x1F7E2;`, `&#x1F7E1;`, `&#x1F7E3;`, `&#x1F7E0;`, `&#x26AB;`, `&#x26AA;`
- common radiology-study icons: `&#x1F9E0;`, `&#x1F511;`, `&#x2696;&#xFE0F;`, `&#x1F4A1;`, `&#x1F9FE;`, `&#x1F4CC;`, `&#x1F4DA;`, `&#x1F4DD;`, `&#x1F50D;`, `&#x1F4D6;`, `&#x1F9EC;`, `&#x1FA7B;`, `&#x1F5BC;&#xFE0F;`, `&#x1F4CA;`, `&#x1F9ED;`, `&#x1F6A9;`, `&#x2705;`, `&#x2B50;`, `&#x26A0;&#xFE0F;`, `&#x1F9EA;`, `&#x1F3F7;&#xFE0F;`, `&#x1F4CD;`, `&#x1F3AF;`, `&#x1F6E0;&#xFE0F;`, `&#x1F52C;`, `&#x1F5C2;&#xFE0F;`, `&#x1F4CE;`, `&#x2733;&#xFE0F;`, `&#x1F536;`, `&#x1F537;`, `&#x2666;&#xFE0F;`, `&#x25AA;&#xFE0F;`

Within Key Imaging Findings:
- group findings by modality when the article supports modality-specific organization
- do not flatten modality-specific findings into one mixed list
- modality subsection titles must be on their own lines with bullets beneath them

Preferred imaging subsections when supported:
- <b>Radiography / Best Initial Test</b>
- <b>CT</b>
- <b>MR Imaging</b>
- <b>Ultrasound / Other</b>
- <b>Distribution / Hallmark Signs</b>

End with:
- a concise <b>Super Summary</b> section

Use bullets generously for readability.
Prefer compact, high-yield bullets rather than long prose bullets.
Each bullet should ideally communicate one radiology takeaway.
Use nested bullets when helpful to split longer concepts into smaller reviewable chunks.
Omit unsupported sections rather than fabricating them.

-----------------------------------------------------
SUMMARY CONTENT REQUIREMENTS

Every bullet must be directly supported by the article text.

The summary should:
- Preserve important qualifiers exactly in meaning
  (e.g., most common, usually, classic, uncommon, rare, often, may)
- Include high-value negatives when the article emphasizes them
- Capture key discriminators, not just isolated facts
- Preserve distinctions between similar entities when the article teaches them separately
- Prefer board-style bullets over long prose, but still remain information-dense
- Prioritize radiology study usefulness, especially imaging findings, high-yield radiology facts, measurements, criteria, scoring systems, differentials, and pitfalls
- Usually contain approximately 10-20 bullets total across sections when the source supports that level of detail
- Elevate article-defined abbreviations or acronyms into a dedicated section when source-supported
- Elevate hallmark signs, favored locations, distributions, measurements, criteria, scoring systems, and diagnostic pivots into dedicated bullets or subsections when source-supported

The summary must NOT:
- Collapse distinct concepts into one vague bullet
- Become a generic mini-cheat-sheet detached from the article's actual structure
- Include management, thresholds, or epidemiology unless explicitly stated in the article
- Add MRI behavior unless explicitly stated in the article
- Add ultrasound behavior unless explicitly stated in the article
- Introduce outside synthesis from Core, Radiopaedia, or external literature
- Collapse modality-specific imaging into mixed prose when the source organizes it by modality
- Use inline heading-plus-sentence formatting when the same content can be rendered as a header with bullets beneath it

-----------------------------------------------------
FAILURE CONDITION

If a card originates from RadPrimer or STATdx and its summary field is:
- missing
- clearly derived from a digest instead of full article review
- overly compressed to the point that key distinctions are lost
- contaminated with Core or outside knowledge

then the card is structurally incomplete and must not be exported.

If a card does NOT originate from RadPrimer/STATdx:
- The summary field must remain blank.

-----------------------------------------------------

Front-Side Blinding Rule (Global):

Fronts must contain ONLY the elements required by their specific template.

No front may include:
- Diagnosis labels (unless template explicitly allows neutral Entity_Label)
- Named signs
- Imaging descriptors
- Differential hints
- Caption-derived findings
- Management clues
- Any form of diagnosis leakage

All fronts must preserve:
- Cognitive integrity
- Pattern-recognition training
- Diagnostic discrimination training
