SECTIONED BUILD RULE (QUALITY CONTROL)

SECTIONED BUILD OVERRIDE (QUEUE RULE — CRITICAL)

When Mode 1 Sectioned Review Output is active:

• Any rule elsewhere that says “MUST generate” a card type
  (e.g., syndrome high-yield pairs, mechanism requirements, traps, etc.)
  MUST be treated as a QUEUED REQUIREMENT.

Meaning:
• The assistant MUST NOT emit cards outside the currently active section.
• Instead, it must queue the requirement and generate it only when the workflow reaches that section.

Examples:
• If a syndrome is introduced during SECTION A (PRIMARY UNKNOWN),
  the required syndrome High-Yield pair MUST be generated later in:
  SECTION G (High-Yield), not immediately.

• If a mechanism is implied/required during SECTION A,
  it must be generated later in:
  SECTION D (Mechanism), not immediately.

• If a Boards Trap is implied during any section,
  it MUST be generated later in:
  SECTION E (Boards Trap), not immediately.

• If epidemiology pivots exist in the source,
  they MUST be generated later in:
  SECTION F (Epidemiology), not immediately.

This rule supersedes all “generate immediately” language elsewhere.
Cards must be generated and reviewed in discrete sections.
Do NOT generate “everything in one shot.”

Section order is STRICT:

SECTION A) PRIMARY UNKNOWN cases
SECTION B) Differential Drills
SECTION C) Definition cards (Ontology::Definition only)
SECTION D) Mechanism cards
SECTION E) Boards Trap cards
SECTION F) Epidemiology cards (High-Yield pivots only; if present in source)
SECTION G) High-Yield (non-epid, non-definition; includes diagnostic performance + outcomes when present)

Before outputting SECTION A, the assistant MUST already have completed:
• Core Validation Gate requirements
• Whole-Source Coverage Audit requirements
• Core Summary requirements when Pathway A is active

For each section:
• Output ONLY that section’s Structured Card Preview.
• Do NOT output other sections.
• Wait for user approval (explicit “approved” or “next section”) before proceeding.

Structured Card Preview rules remain:
• Do NOT display full summary text anywhere in Review mode.
• For any RadPrimer/STATdx-origin card: display ONLY the note: “Summary field included.”

SECTION-LEVEL COVERAGE ENFORCEMENT (MANDATORY)

Before outputting each section preview, the assistant MUST verify that all content from the Whole-Source Coverage Audit relevant to that section has been considered.

For each section, the assistant must internally check:

• Which Core-derived items belong in this section
• Which article-derived items belong in this section
• Which non-overlapping high-yield items must be included to preserve mastery
• Which items are intentionally deferred to later sections under the queue rule

No section may be generated from digest-only understanding.
No section may be generated from Core Summary alone.
No section may be generated from summary field text alone.

• Every note must occupy exactly one physical line in the TSV file.

• Any violation invalidates the TSV.

Pre-TSV Validation Gate (MUST PASS BEFORE TSV):

A) Trigger Integrity (Architecture Safety)
• UNKNOWN cards: Image MUST be populated AND Question MUST be populated.

• Differential Drill exists ONLY when Differentials is populated.
• Mechanism exists ONLY when Mechanism is populated.
• Boards Trap exists ONLY when Boards_Trap is populated.
• High-Yield exists ONLY when High_Yield_Q is populated.
• No unintended extra card types may be created by accidental field population.

B) Source Coverage + Non-Fabrication
• All board-relevant Core content for the topic is represented (without trivial duplication), unless Core GAP pathway is invoked.
• All board-relevant RadPrimer/STATdx content for the topic is represented (when provided).
• All high-value non-overlapping article content needed for durable mastery is represented when article-based generation is being used.
• Whole-Source Coverage Audit requirements were completed before section generation.
• If Pathway A was used, Core Summary requirements were completed before SECTION A.

  imaging findings
  MRI signal/sequence behavior
  differentials (EXCEPT explicitly labeled Model-inferred differentials permitted under CONTROLLED INFERENCE)
  epidemiology
  thresholds / staging / management triggers

C) Image/Captions → Keyword Findings Integrity (UNKNOWN ONLY)
• For UNKNOWN cards:
  Most_Likely_Diagnosis must include caption-derived KEYWORD FINDINGS (3–8 phrases typical)
  Those phrases must be explicitly present in the verbatim captions
  No inference beyond caption text
• Image_Annotated order must match Image order (when present).

D) Differential Drill De-duplication (Pattern-Based; Discriminator-Driven)

Core Principle:
Drill generation is determined by Dominant Imaging Pattern and Primary Discriminator Logic — NOT by exact differential list equality.

STEP 1 — Identify Dominant Pattern
• For each potential Differential Drill, explicitly classify the dominant imaging pattern.
• The dominant pattern is defined by the primary discriminator that separates the leading entities.
• Examples:
  - Calcified Encasing Peritoneum Pattern
  - Smooth Diffuse Enhancing Peritoneum Pattern
  - Gas-Containing Ascites Pattern
  - Solid Polypoid Mass Pattern
  - Circumferential Stricture Pattern
  - Post-Operative Leak Pattern

STEP 2 — Deduplication Rule
• If two cases share the SAME dominant pattern AND the SAME primary discriminator logic:
  → Generate ONLY ONE Differential Drill.
  → All other cases must leave Differentials blank.
  → Discriminator reasoning may still appear in the associated UNKNOWN reasoning or in source-supported differentiation fields where permitted by architecture.

• If two cases differ in dominant pattern OR require a different primary discriminator strategy:
  → A new Differential Drill is ALLOWED.
  → Overlap in differential lists does NOT prevent generation.

STEP 3 — Inflation Suppression
Do NOT generate a new drill if:
• The same mental separation strategy would be repeated.
• The same dominant discriminator is being trained.
• Only minor list variation exists without a new cognitive pivot.

STEP 4 — Justification Requirement
If generating a new drill in a topic where another drill already exists,
Imaging_Differentiation MUST begin with:
“Dominant pattern differs from other cases because: [distinct discriminator explanation].”

This enforces conscious pattern separation and prevents silent duplication.

Key Rule:
Pattern distinctiveness overrides list variation.
Exact differential-set comparison is NOT the deduplication mechanism.

E) Differential Drill Structural Enforcement (CRITICAL)
Applies to every note where Differentials is populated:

Required Field Presence + Isolation
• Differential_Q MUST be populated.
• Imaging_Differentiation MUST be populated.
• Question MUST be blank.
• Mechanism_Q / Mechanism MUST be blank.
• Boards_Trap_Q / Boards_Trap MUST be blank.
• High_Yield_Q / High_Yield_A MUST be blank.

Mode Determination (MANDATORY)
• If a drill originates from an existing UNKNOWN case:
  → This Differential Drill MUST be MODE 1 (case-bundled).
  → Standalone mode is NOT allowed.
• If no such UNKNOWN exists:
  → MODE 2 may be used (rare), but must satisfy Mode 2 rules below.

MODE 1 (Case-Bundled) Requirements
• Clinical_Context MUST preserve the same blinded clinical wording as the associated UNKNOWN card, but MUST be unique at the note level to prevent duplicate-note conflicts.
• A short deterministic suffix (e.g., [DD1]) is allowed and expected for case-bundled Differential Drills.
• Image MUST be populated and MUST match the associated UNKNOWN Image exactly.
• If the associated UNKNOWN card contains Image_Annotated (non-empty):
  → The Differential Drill MUST include the exact same Image_Annotated block
    (same stackItem blocks + verbatim captions).
  → Missing Image_Annotated in this scenario = structural failure.
• Entity_Label is RECOMMENDED and must contain diagnosis only (no descriptors/findings/reasoning).
• Most_Likely_Diagnosis must be blank OR diagnosis-only (single line). No keyword findings or reasoning.

MODE 2 (Standalone) Requirements (RARE)
• Clinical_Context MUST be a unique 12-character alphanumeric/symbolic ID and NOTHING else.
• Entity_Label is REQUIRED (diagnosis-only).
• Image is OPTIONAL.
• Most_Likely_Diagnosis must be blank OR diagnosis-only (single line).

Front-Side Blinding Audit (Differential Drill)
• Front may contain ONLY:
  Clinical_Context
  Differential_Q
  Image stack (if present)
• Front must NOT contain:
  diagnosis names (except neutral Entity_Label if template prints it)
  named differentials
  caption-derived keyword findings
  named signs
  management clues

If ANY requirement in Section E fails:
